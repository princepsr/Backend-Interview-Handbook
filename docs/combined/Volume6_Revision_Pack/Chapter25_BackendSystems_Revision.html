Section 1: REST APIs

Reading guide: focus on WHY each constraint or rule exists — interviewers probe the reasoning, not just the label.

Q1: What are the 6 REST architectural constraints, and why does each one matter? Concept Check

One-line answer: REST is defined by six constraints — Client-Server, Stateless, Cacheable, Uniform Interface, Layered System, and Code on Demand — each one targeting a specific scalability or maintainability concern.

Full answer: I think of the six REST constraints as a design checklist, not just a trivia list. The first is Client-Server separation: the UI is decoupled from data storage, so either side can evolve independently — you can rewrite your React frontend without touching the API. The second is Stateless: every request must carry all the information the server needs to fulfil it — no session state lives on the server between requests — which means any server instance can handle any request, enabling horizontal scaling. Third is Cacheability: responses must declare whether they can be cached using Cache-Control headers, so CDNs and browsers can avoid redundant network calls. Fourth is Uniform Interface, which has four sub-constraints: resources are identified by URIs, clients manipulate resources through their representations (JSON/XML), messages are self-descriptive (Content-Type headers tell you how to parse them), and HATEOAS means responses include links to next available actions. Fifth is Layered System: the client cannot tell whether it is talking to the origin server or an intermediary like a load balancer, CDN, or proxy — this enables transparent infrastructure changes. Sixth is Code on Demand, the only optional constraint: the server can send executable code to the client, which is how JavaScript is delivered to browsers. In interviews I always connect each constraint back to a concrete scalability or maintainability benefit.

Lead with “stateless enables horizontal scaling” — that is the constraint interviewers most want you to explain deeply.

Gotcha follow-up: Which constraint do most real-world APIs violate, and is that acceptable? Most real-world APIs violate HATEOAS — the Level 3 constraint of the Richardson Maturity Model — because implementing it fully adds significant complexity for often marginal client benefit. In practice, clients still hardcode URIs rather than discovering them from response links. This is a pragmatic trade-off: the industry has largely settled on Level 2 REST (resources + correct HTTP verbs + status codes) as “good enough,” and HATEOAS is reserved for hypermedia-heavy use cases like HAL or JSON:API. I would argue violating HATEOAS is acceptable as long as the team is deliberate about it and documents the API contract through OpenAPI instead.

Q2: What is the difference between HTTP method safety and idempotency, and why does it matter for retry logic? Concept Check

One-line answer: Safe methods have no server-side side effects; idempotent methods produce the same result no matter how many times they are called — and idempotency is what makes automatic retries safe.

Full answer: Safety and idempotency are two separate properties, and it is important not to conflate them. A method is safe if it does not modify server state — GET, HEAD, and OPTIONS are safe because they are read-only operations. A method is idempotent if calling it N times produces the same outcome as calling it once — GET, HEAD, OPTIONS, PUT, and DELETE are all idempotent. PUT is idempotent because sending the same update twice results in the same final resource state. DELETE is idempotent because deleting an already-deleted resource (returning 404 on the second call) still leaves the system in the same state. POST is neither safe nor idempotent — each POST typically creates a new resource, so sending it twice creates two records. PATCH is interesting: it is not guaranteed idempotent by the spec — “set field to value” is idempotent, but “increment counter by 1” is not. Why does this matter in practice? When a network timeout occurs, infrastructure like service meshes, API gateways, and load balancers will automatically retry idempotent requests — they know it is safe to do so. For POST, you must implement an Idempotency-Key pattern manually to make retries safe.

Use a concrete example: “PUT /users/1 with the same body twice leaves the same user — safe to retry. POST /orders twice creates two orders — dangerous to retry.”

Gotcha follow-up: Is DELETE truly idempotent if the server returns 404 on the second call? Yes — idempotency is about the state of the server, not the HTTP status code of the response. After calling DELETE /users/1 twice, the user does not exist in either case — the server state is identical. The second call returns 404, but the outcome is the same: the resource is gone. Some strict implementations return 200 or 204 on both calls to avoid confusing clients, but the RFC permits 404. The key insight is that idempotency is a guarantee about side effects on the system, not about response uniformity.

Q3: Walk me through the HTTP status codes an interviewer is most likely to ask about. Concept Check

One-line answer: The codes that trip people up are 401 vs 403, 400 vs 422, 201 vs 204, and 502 vs 503 — each has a precise meaning that maps to a specific failure mode.

Full answer: I group status codes by what they communicate. In the 2xx range: 200 OK means success with a response body, used for GET/PUT/PATCH. 201 Created means a new resource was created — always include a Location header pointing to the new resource URI. 204 No Content means success with no body, used for DELETE and sometimes PUT. In the 3xx range: 301 Moved Permanently tells browsers to cache the redirect permanently, while 302 Found is a temporary redirect that must not be cached. The 4xx client errors are where interviewers focus: 400 Bad Request covers malformed syntax or missing required parameters. 401 Unauthorized is a misleadingly named code — it actually means “not authenticated,” as in the client has not logged in or provided a valid token. 403 Forbidden means the client is authenticated but not authorized — they are logged in but lack permission for this resource. 404 Not Found means the resource does not exist. 409 Conflict covers state conflicts like duplicate keys or optimistic locking failures. 422 Unprocessable Entity is for semantically invalid requests — the format is correct JSON but the business rules are violated. 429 Too Many Requests means rate limit exceeded — always include a Retry-After header. In the 5xx range: 500 is a generic server failure, 502 Bad Gateway means an upstream service returned an invalid response, and 503 Service Unavailable means the server is down or overloaded — include Retry-After.

Always distinguish 401 (not authenticated) from 403 (not authorized) — mixing them up is a red flag to interviewers.

Gotcha follow-up: Should you return 404 or 403 when an authenticated user tries to access a resource that exists but belongs to another user? This is a security trade-off. Returning 403 Forbidden is technically correct — the resource exists but the user lacks permission. However, returning 404 Not Found is the more secure choice in many contexts because it does not leak the existence of the resource to an attacker. If an attacker does not know whether resource ID 12345 exists, they cannot enumerate resources by probing for 403s. This is called resource existence leakage prevention. The right answer depends on sensitivity: for a social media profile it might be fine to return 403, but for medical records or financial data, 404 is safer.

Q4: What are the API versioning strategies, and when would you choose each? Tradeoff Question

One-line answer: URI path versioning (/api/v1/users) is the most pragmatic for public APIs; header versioning is cleaner for internal APIs — the choice depends on visibility, cacheability, and audience.

Full answer: There are four main strategies. URI path versioning — /api/v1/users — puts the version directly in the URL. It is the most visible, easiest to document, easy to bookmark, and simple to test in a browser. The downside is that it violates REST purity: a version is metadata about the representation, not part of the resource identity. But I would argue pragmatism wins over purity for public APIs. Query parameter versioning — /api/users?version=1 — is simple to implement but pollutes the query string and is easy to miss or forget. Custom header versioning — Accept-Version: v1 or X-API-Version: 1 — keeps URLs clean and is better for internal microservice APIs. Content negotiation — Accept: application/vnd.myapi.v1+json — is the RESTful purist approach: the client declares what representation format it accepts. It is the most technically correct but hardest to use in practice because most HTTP clients handle Accept headers awkwardly. My recommendation: use URI versioning for public APIs consumed by third parties — it is the most visible and interoperable. Use header versioning for internal service-to-service APIs where the client is controlled. Never use content negotiation unless you are building a hypermedia API for advanced clients.

Frame your answer as a decision with context: “I would choose X for public APIs because of Y, and Z for internal because of W.”

Gotcha follow-up: How do you handle deprecating an old API version without breaking clients? I follow a sunset policy: add a Sunset header (an RFC 8594 standard) to responses from the deprecated version, giving a specific date when it will be removed — for example, Sunset: Sat, 01 Jan 2026 00:00:00 GMT. I also add a Deprecation header and a Link header pointing to the migration guide. I communicate the deprecation through changelog, email, and developer portal announcements. I monitor traffic on the old version using API Gateway logs to identify which clients are still calling it. I set a minimum notice period — typically 6 to 12 months for external APIs. Only after traffic drops to zero or a contractual deadline passes do I remove the endpoint. Never hard-delete without a sunset period for public APIs.

Q5: Explain the Idempotency Key pattern — how does it work, and when is it critical? Design Scenario

One-line answer: The client generates a UUID per logical operation and sends it as a header; the server caches the result keyed on it, so retries return the cached response instead of reprocessing.

Full answer: The Idempotency Key pattern solves the double-execution problem for non-idempotent operations. Here is how it works in detail: the client generates a UUID (for example, using UUID v4) before making the request and sends it in the Idempotency-Key: header. The server, on receiving the request, first checks a storage layer — typically Redis with a TTL of 24 hours — to see if it has already processed a request with this key. If it has, it returns the previously computed response immediately without re-executing the business logic. If it has not, it processes the request, stores the result keyed on the idempotency key, and then returns the response. This is critical for: payment processing (double-clicking “Pay” should not charge twice), order creation (network retry should not create duplicate orders), and any POST that causes irreversible side effects. An important detail: on a replay, return 200 OK, not 201 Created, to signal to the client that this is a cached response and no new resource was created. Store both the response status code and body so the replay is byte-for-byte identical to the original response. The UUID must be client-generated so the client owns the idempotency guarantee — if the server generated it, a client could not safely retry after a timeout.

Emphasize the storage layer design: “I use Redis with a TTL so the key expires after a reasonable window — 24 hours for payments.”

Gotcha follow-up: What happens if two concurrent requests arrive with the same idempotency key before either has been processed? This is the race condition case, and it requires a distributed lock. When the first request arrives, the server should acquire an atomic lock on the idempotency key — using Redis SETNX or a SET NX PX operation — before starting processing. If the second concurrent request arrives with the same key and finds the lock already held, it should either wait briefly and retry, or return 409 Conflict with a message saying the operation is in progress. Without this lock, two concurrent requests with the same key could both pass the “not found” check and both execute, defeating the purpose of the idempotency key entirely.

Q6: What is RFC 7807, and why is it better than a custom error response format? Concept Check

One-line answer: RFC 7807 defines a standard JSON error response shape with Content-Type: application/problem+json, so every client can handle errors from any API without learning a bespoke schema.

Full answer: RFC 7807, titled “Problem Details for HTTP APIs,” standardizes what error response bodies look like. Before it existed, every API team invented their own error format: some used { “error”: “…” }, others used { “message”: “…”, “code”: 42 }, and clients had to learn each one separately. RFC 7807 defines five standard fields: type is a URI that identifies the error type and should link to documentation — for example, https://api.example.com/problems/insufficient-balance; title is a short, human-readable summary like “Insufficient Balance” that is stable across occurrences; status mirrors the HTTP status code in the body for convenience; detail is a human-readable explanation of this specific occurrence, like “Your balance is $30 but the charge requires $50”; and instance is a URI that identifies this specific error event, useful for support ticket correlation. The Content-Type must be application/problem+json. You can extend it with domain-specific fields — add “balance”: 30 and “required”: 50 to the example above. The benefits: clients can write generic error handling logic once, documentation tooling understands the format, and debugging is dramatically easier because error types are stable URI-identified concepts rather than ephemeral string messages.

In Spring Boot, use ProblemDetail (Spring 6+) which natively implements RFC 7807 — mention it to show production awareness.

Gotcha follow-up: Should the type URI in RFC 7807 resolve to an actual web page? Yes, ideally it should — RFC 7807 specifies that the type URI should resolve to human-readable documentation describing the error type. When a developer copies the URI into a browser, they should land on a page explaining what caused the error and how to fix it. In practice, many teams use a URI format like https://errors.example.com/insufficient-balance and either serve a real page there or accept that the URI serves as a stable identifier rather than a live link. The spec explicitly states it is acceptable to use the URI as an opaque identifier if resolving it is not practical, but the best practice is to make it resolvable for developer experience.

Q7: Compare cursor-based and offset-based pagination — why does offset fail at scale? Tradeoff Question

One-line answer: Offset pagination scans and discards rows from the start of the table — O(offset) cost — and suffers from page drift on concurrent writes; cursor pagination is an O(1) index seek with no drift.

Full answer: Offset pagination uses SQL’s OFFSET clause: SELECT * FROM posts ORDER BY created_at DESC OFFSET 100 LIMIT 20. This tells the database to scan 120 rows and discard the first 100 before returning 20. At OFFSET 1000, the database scans 1020 rows and discards 1000 — pure waste. At OFFSET 100,000, you are scanning 100,020 rows per page request. This is O(offset) cost, and it gets worse as users page deeper. There is also a correctness problem: if a new post is inserted while a user is paginating, every subsequent page shifts by one row — the user skips a post or sees a duplicate. Cursor pagination solves both problems. The cursor encodes the sort key of the last-seen item — for example, the post_id or created_at of the last item on the current page. The next-page query becomes WHERE id < cursor ORDER BY id DESC LIMIT 20 — this is a direct B-tree index seek, O(1) regardless of how deep into the dataset you are. Concurrent inserts do not affect it because the query anchors to a specific key value, not a row count. The trade-off: you lose random page access — you cannot jump directly to page 47. For feeds, search results, and large datasets, cursor pagination is always the correct choice. Offset pagination is acceptable only for small admin tables where deep pagination never happens.

Always mention the concurrent-write page drift problem — interviewers love it because it is a subtle correctness bug, not just a performance issue.

Gotcha follow-up: How do you encode the cursor to prevent clients from tampering with it? I base64-encode the cursor value before sending it to the client, which obscures the implementation detail — the client does not know whether it is a timestamp, an ID, or a composite key. For additional security on sensitive APIs, I sign the cursor with an HMAC using a server-side secret key, and validate the signature on receipt. This prevents a client from crafting a malicious cursor value that could inject SQL or expose internal data. The cursor should be treated as an opaque token by the client — they must not parse it or construct their own.

Q8: Explain the four main rate limiting algorithms and their trade-offs. Concept Check

One-line answer: Token Bucket allows controlled bursting and is the production standard; Fixed Window is simple but vulnerable to boundary bursts; Sliding Window Log is exact but memory-heavy; Sliding Window Counter approximates well at low memory cost.

Full answer: Token Bucket is the algorithm I reach for in production. A bucket holds up to N tokens and refills at a constant rate R per second. Each incoming request consumes one token. If the bucket is full and no requests have come in, tokens accumulate up to the bucket capacity — allowing a burst of up to N requests after a quiet period. If the bucket is empty, the request is rejected with 429. This models real-world traffic well: a user can burst briefly but cannot sustain above the average rate. Fixed Window Counter counts requests per fixed time window — for example, 100 requests per minute. It is trivially simple to implement with Redis INCR and EXPIRE. The critical flaw: at the boundary between two windows, a client can send 100 requests at 11:59:59 and 100 more at 12:00:01 — getting 200 requests in 2 seconds despite a “100 per minute” limit. This is the boundary burst problem. Sliding Window Log solves the boundary burst by storing the exact timestamp of every request in a sorted set, then counting requests in the past 60 seconds. This is perfectly accurate but uses O(requests) memory per client — expensive at scale. Sliding Window Counter is the practical middle ground: it uses two consecutive fixed windows with a weighted interpolation to approximate how many requests occurred in the sliding window. Low memory, much better accuracy than fixed window, slight approximation error. For most production use cases: Token Bucket for client-facing APIs where burst tolerance is desirable, Sliding Window Counter for strict rate enforcement.

The Fixed Window boundary burst is a classic interview trick question — always name it explicitly.

Gotcha follow-up: Where do you store rate limit counters in a distributed system with 10 API gateway instances? The counters must be stored in a shared external store — Redis is the standard choice. Each API gateway instance reads and increments the same Redis key for a given client. Use Redis’s atomic INCR command combined with EXPIRE to ensure the counter is both incremented and expired atomically. For Token Bucket, use a Lua script to atomically check the token count, refill based on elapsed time, and decrement — all in one atomic operation, avoiding race conditions. Without a shared store, each gateway instance has its own counter and a client can make 10x the allowed requests by round-robining across instances.

Q9: How do ETags and conditional requests work, and what two problems do they solve? Concept Check

One-line answer: ETags are resource fingerprints that enable bandwidth-saving conditional GETs (If-None-Match) and optimistic locking on updates (If-Match) — solving both network efficiency and concurrent modification conflicts.

Full answer: An ETag, or Entity Tag, is a fingerprint — typically an MD5 or SHA hash of the resource’s content — that the server includes in every response: ETag: “abc123”. It serves two distinct purposes. The first is bandwidth optimization: when a client re-fetches a resource it already has, it sends If-None-Match: “abc123” in the request. If the resource has not changed since the client last fetched it, the server responds with 304 Not Modified with no body — the client uses its cached copy. This saves bandwidth proportional to the response size, which matters for large resources like product catalogs or user profiles. The second purpose is optimistic locking: when a client wants to update a resource, it includes If-Match: “abc123” in the PUT or PATCH request. The server compares the tag against the current resource state. If another client has modified the resource in the meantime — changing the ETag — the server responds with 412 Precondition Failed, telling the client to re-fetch and re-apply their change on top of the latest version. This prevents the “lost update” problem: two clients read version 1, both modify it, and the last writer silently overwrites the first writer’s changes. ETags are essentially HTTP-level optimistic locking, equivalent to a version column in a database row.

Distinguish the two use cases clearly: If-None-Match for caching (read), If-Match for concurrent update protection (write).

Gotcha follow-up: Should ETags be strong or weak, and what is the difference? A strong ETag — the default, quoted string without a prefix — means byte-for-byte identity: two responses with the same strong ETag must have identical bodies. A weak ETag — prefixed with W/, like W/“abc123” — means semantic equivalence: the representations are equivalent for the purpose of the application even if bytes differ slightly (for example, different gzip compression runs). Strong ETags are required for safe use with If-Match conditional updates — you need byte-for-byte guarantees for optimistic locking. Weak ETags are sufficient for caching with If-None-Match because you only care whether the content is semantically the same. In practice I use strong ETags for all PUT/PATCH operations and can use weak ETags for GET caching when byte-perfect equality is not required.

Q10: Explain CORS and the preflight mechanism — why does the browser do it? Concept Check

One-line answer: CORS is a browser security mechanism that prevents malicious scripts from making credentialed cross-origin requests; the preflight OPTIONS request checks server permissions before the actual request is sent.

Full answer: CORS, which stands for Cross-Origin Resource Sharing, is a browser enforcement mechanism rooted in the Same-Origin Policy — the rule that scripts on one origin (scheme + domain + port) cannot read responses from a different origin. Without this policy, a malicious site could use JavaScript to call your bank’s API using your cookies and read your account balance. The browser enforces this by intercepting cross-origin requests. For “non-simple” requests — those using methods other than GET/POST, or headers beyond a small safe list — the browser first sends an OPTIONS preflight request to the target origin. This preflight includes the Origin header, the Access-Control-Request-Method (what HTTP method the real request will use), and Access-Control-Request-Headers (what custom headers it will send). The server inspects these and responds with Access-Control-Allow-Origin (which origins are permitted), Access-Control-Allow-Methods, Access-Control-Allow-Headers, and optionally Access-Control-Allow-Credentials: true if cookies should be included. Only if the server explicitly grants permission does the browser proceed with the real request. Access-Control-Max-Age caches the preflight response for a specified number of seconds, preventing a round-trip preflight before every single request. Simple requests — GET or POST with standard headers — skip the preflight but still have their response blocked by the browser unless the server includes the Allow-Origin header. Critically: CORS is enforced by the browser, not the server — a direct curl request is never blocked by CORS.

Always end with the key insight: “CORS is a browser security feature — it only affects browser JavaScript, never server-to-server calls.”

Gotcha follow-up: If your API returns Access-Control-Allow-Origin: * can it also return Access-Control-Allow-Credentials: true? No — this combination is explicitly prohibited by the CORS specification. Wildcard origin (*) with credentials: true would mean any website could make credentialed requests (sending cookies or Authorization headers) to your API, which defeats the entire purpose of CORS protection. Browsers will refuse to accept such a response and will block the request. To allow credentialed cross-origin requests, you must specify an exact origin — for example, Access-Control-Allow-Origin: https://app.example.com — rather than the wildcard. This requires your server to dynamically set the Allow-Origin header based on the incoming Origin header, after validating it against an allowlist.

Q11: Compare REST, GraphQL, and gRPC — when would you choose each? Tradeoff Question

One-line answer: REST is the default for public APIs with HTTP caching needs; GraphQL solves over/under-fetching for complex frontends; gRPC delivers low-latency binary communication for internal microservices.

Full answer: These three are not competing standards — they solve different problems. REST is the industry default: it maps naturally to HTTP semantics, benefits from native HTTP caching through Cache-Control and ETags, is universally understood, and has the broadest tooling ecosystem. I choose REST for any public API where third-party developers need to consume it, because the learning curve is minimal. The weaknesses are over-fetching — the client receives fields it does not need — and under-fetching — a single view requires multiple sequential API calls. GraphQL addresses both of these problems: the client writes a query specifying exactly which fields it needs, and the server returns precisely that shape. This is powerful for complex frontends like dashboards that need data from many entities in a single request, or for BFF (Backend for Frontend) patterns where the gateway aggregates microservice data per client. The trade-offs: GraphQL queries are typically POST requests, which disables HTTP caching — you must implement application-level caching. It is also prone to the N+1 problem in resolvers, which requires DataLoader batching. gRPC uses Protocol Buffers — a compact binary serialization format — over HTTP/2, which supports multiplexed streams and has lower latency than HTTP/1.1. It is strongly typed with auto-generated client and server stubs in any language from a .proto file. I choose gRPC for high-frequency, latency-sensitive internal microservice communication — for example, a checkout service calling inventory service hundreds of times per second. The downside: binary format is not human-readable, and browser support requires a gRPC-Web proxy.

Structure your answer as a decision tree: “I default to REST, I switch to GraphQL when I have over/under-fetching problems, I switch to gRPC when I need maximum internal throughput.”

Gotcha follow-up: How do you handle caching with GraphQL given that queries are POST requests? Since GraphQL queries go over POST, they are excluded from standard HTTP cache mechanisms (CDNs and browsers only cache GET). There are three approaches: first, use GET requests for queries — GraphQL allows passing the query as a URL query parameter, enabling CDN caching with GET semantics for read-only queries. Second, use Persisted Queries — the client and server pre-agree on a map of query hashes to query strings; the client sends only the hash, which is short enough to fit in a GET URL. This is what Apollo’s Automatic Persisted Queries implements. Third, use application-level caching in the GraphQL layer itself — cache resolver results in Redis keyed on the query hash and arguments, with appropriate TTLs per entity type.

Q12: What does an API Gateway do, and what is the BFF pattern? Design Scenario

One-line answer: An API Gateway is the single entry point handling routing, auth, and cross-cutting concerns; a BFF is a purpose-built gateway per client type that aggregates and shapes data for that specific client’s needs.

Full answer: An API Gateway sits at the edge of your system and is the single point through which all client traffic enters. It handles infrastructure-level concerns: routing — inspecting the path and forwarding to the appropriate microservice; SSL termination — decrypting HTTPS once at the gateway so internal services communicate over plain HTTP on the private network; authentication — validating JWT tokens or OAuth2 access tokens before the request reaches any downstream service; rate limiting — enforcing per-client or per-endpoint request quotas; request and response transformation — adding headers like X-Request-Id, removing sensitive internal headers before responding to the client, or translating protocols; and logging and distributed tracing — injecting trace IDs so the full request journey is observable. Examples include AWS API Gateway, Kong, NGINX, and Spring Cloud Gateway. The critical design rule: never put business logic in the API Gateway. It should be pure infrastructure. The BFF (Backend for Frontend) pattern extends this idea by creating a separate, application-level gateway for each client type. The web BFF and mobile BFF are different services, each owned by their respective frontend team. The mobile BFF returns compressed, minimal JSON suited for slow connections and small screens; the web BFF returns richer responses. The BFF also aggregates data from multiple microservices into a single client-optimized response — something an infrastructure-level gateway should never do.

The key distinction: API Gateway is infrastructure (no business logic), BFF is application-level (client-specific aggregation).

Gotcha follow-up: What are the failure mode risks of routing all traffic through a single API Gateway? The API Gateway is a single point of failure and a potential performance bottleneck. Mitigations: deploy multiple gateway instances behind a hardware or cloud load balancer; use an active-active multi-region setup for global APIs; ensure the gateway is stateless so any instance can handle any request. Circuit breakers at the gateway level prevent a slow downstream service from exhausting gateway threads. For the performance bottleneck: the gateway should do minimal processing — offload JWT validation to a fast in-memory operation using the public key, not a network call to the authorization server. Profile gateway latency separately from service latency so you can detect gateway overhead independently.

Q13: How does OpenAPI benefit a team beyond just documentation? Concept Check

One-line answer: OpenAPI is a machine-readable contract that enables auto-generated client SDKs, mock servers for parallel frontend development, and contract testing to catch breaking changes before they reach production.

Full answer: OpenAPI, formerly called Swagger, is a YAML or JSON specification that formally describes a REST API’s endpoints, request/response schemas, authentication, and error responses. In Spring Boot, springdoc-openapi generates the spec automatically from annotations like @Operation, @ApiResponse, and @Schema on your controllers and DTOs — the spec stays in sync with the code. The benefits go well beyond a Swagger UI documentation page. First: auto-generated client SDKs. The openapi-generator tool can generate a fully typed API client in Java, TypeScript, Python, Go, or any of 50+ languages from the spec — no manual HTTP calls, no JSON parsing boilerplate. Second: mock servers. Tools like Prism or WireMock can serve a mock API that returns realistic example responses based purely on the spec, before a single line of backend code is written. This enables frontend and backend teams to develop in parallel against the same contract. Third: contract testing. Tools like Dredd or Pact validate that the running server’s actual responses conform to the spec — catching breaking changes automatically in CI before they reach consumers. Fourth: the spec enables API linting (Spectral) to enforce style rules like “all endpoints must have a description” or “all error responses must use RFC 7807 format.”

Mention mock servers explicitly — the ability to give the front-end team a mock API on day one is the most tangible productivity benefit.

Gotcha follow-up: What is the difference between code-first and design-first OpenAPI, and which is better? Code-first means you write the implementation and generate the OpenAPI spec from annotations on the code — this is what springdoc-openapi does. Design-first means you write the OpenAPI spec first as a contract, then generate server stubs and client SDKs from it, and implement the stubs. Design-first is the better practice for APIs consumed by external teams: you agree on the contract before implementation, so clients can start their mock-server development in parallel from day one. Code-first is faster for internal or prototype APIs. The risk of code-first is that the spec drifts from the actual implementation if developers forget to update annotations — contract testing in CI catches this drift.

Q14: What is HATEOAS, and why is it rarely fully implemented? Concept Check

One-line answer: HATEOAS means API responses include links to next available actions so clients discover the API at runtime rather than hardcoding URIs — it is Level 3 of the Richardson Maturity Model but is rarely implemented due to complexity vs. benefit trade-offs.

Full answer: HATEOAS stands for Hypermedia As The Engine Of Application State, and it is the fourth sub-constraint of REST’s Uniform Interface. The idea is that a REST client starts with a single known entry-point URL and discovers all other operations from links embedded in responses — exactly like how a browser navigates the web by following anchor tags rather than knowing all URLs in advance. In practice: a GET /orders/123 response not only returns the order data but also includes a links array containing a link to cancel the order, a link to the order’s payment, a link to the shipping resource, and a self link. The client reads these links and follows them based on their rel (relation type) label — it never constructs URLs itself. This decouples the client from server URL structure completely: the server can change all its URLs without breaking clients, because clients always discover URLs from responses. The Richardson Maturity Model defines Level 0 (single endpoint, RPC over HTTP), Level 1 (resource URIs), Level 2 (correct HTTP verbs and status codes), and Level 3 (HATEOAS). Most production APIs stop at Level 2. HATEOAS is rarely fully implemented because: generating correct, contextual link sets for every response requires significant server-side logic; clients must implement link-following logic instead of simple URL construction; and the benefit — client URL independence — is usually managed through versioning and documentation instead. The Spring HATEOAS library provides EntityModel and CollectionModel to implement it when needed.

Be honest about the pragmatics: “HATEOAS is theoretically correct but Level 2 is the industry standard — I would implement HATEOAS only if clients genuinely need to navigate the API without prior URL knowledge.”

Gotcha follow-up: What is the difference between HATEOAS and an API that just adds a links field to its responses? True HATEOAS means the client is driven entirely by hypermedia: it does not hardcode any URLs and does not know what operations are available without reading the response links. A rel attribute on each link communicates what the link does — for example, rel=“cancel” or rel=“payment” — and the client has logic to follow links by their relation type. Simply adding a links field with some URLs is cosmetic HATEOAS: the client still hardcodes which link rels to look for, still has to know the meaning of each rel in advance, and the API is not genuinely self-describing. The test for real HATEOAS: could a generic client navigate your API knowing only the entry-point URL and the media type semantics? If the answer is no — if the client needs out-of-band knowledge about available operations — it is not true HATEOAS.

Q15: How does HTTP content negotiation work? Concept Check

One-line answer: The client sends an Accept header listing acceptable formats with quality weights; the server picks the best match it supports and responds with Content-Type confirming the chosen format.

Full answer: Content negotiation allows a single endpoint to serve multiple response formats without separate URLs. The client sends an Accept header listing the formats it can handle, with optional quality weights (q values) indicating preference order: Accept: application/json, application/xml;q=0.9, /;q=0.8. This means “I prefer JSON, I’ll accept XML with lower preference, and I’ll accept anything else as a last resort.” The server evaluates this against the formats it can produce for that endpoint and selects the highest-quality match. It responds with Content-Type: application/json confirming the chosen format. If the server cannot satisfy any of the client’s accepted formats, it returns 406 Not Acceptable. In Spring MVC, you restrict what formats an endpoint produces using produces = “application/json” in @RequestMapping — or list multiple: produces = {“application/json”, “application/xml”}. The Accept-Language header works the same way for language negotiation: Accept-Language: en-US, fr;q=0.8 — the server picks the best available locale. Content negotiation decouples clients from specific formats: you can add Protobuf or MessagePack support to an existing endpoint without creating new URLs, and old clients that only send Accept: application/json are completely unaffected. The Content-Type header on requests tells the server what format the request body is in — distinct from Accept, which is about the response format.

Connect Accept (what I want back) to Content-Type (what I am sending) — candidates frequently confuse the two headers.

Gotcha follow-up: What happens if a client sends Accept: application/json but the endpoint only produces text/html? The server returns 406 Not Acceptable — it cannot produce a response in any format the client listed as acceptable. In Spring, this is handled automatically if you declare produces = “text/html” on the controller method: Spring will return 406 with no body if the client’s Accept header has no intersection with the declared produces types. A well-behaved API should also provide a body in some format explaining what formats are available — though the 406 response technically has no body in the spec. Some APIs add a Vary: Accept header to successful responses, telling caches that the response content varies by the Accept header value so caches store separate responses per format.

Common Mistakes:

  • Confusing 401 and 403 → 401 means not authenticated (no valid identity); 403 means not authorized (identity confirmed, permission denied). Mixing them up signals shallow HTTP knowledge.
  • Calling POST idempotent → POST creates a new resource on every call. If you mean to make it safe to retry, you need an explicit Idempotency-Key pattern — the method itself gives you no guarantee.
  • Using OFFSET pagination on large tables → at large offsets, the database scans and discards thousands of rows. Switch to cursor-based pagination using WHERE id > cursor for any dataset over a few thousand rows.
  • Returning 200 on a replay of an idempotency-keyed request instead of signaling it correctly → always return 200 (not 201) on replay so the client knows no new resource was created.
  • Wildcard CORS with credentials → Access-Control-Allow-Origin: * and Access-Control-Allow-Credentials: true is rejected by browsers. You must specify an exact origin for credentialed requests.
  • Putting business logic in the API Gateway → the gateway is infrastructure. Business logic belongs in services. A gateway that starts making business decisions becomes a maintenance nightmare.

Quick Revision: REST = stateless + uniform interface + cacheable; safe = read-only; idempotent = same result N times; cursor beats offset at scale; ETags = HTTP optimistic locking.

Section 2: Microservices

Reading guide: focus on the failure patterns and why each resilience tool exists — interviewers probe what goes wrong, not just what the tool is called.

Q1: When should you NOT use microservices? Tradeoff Question

One-line answer: Avoid microservices when your team is small, your domain boundaries are unclear, or your operational maturity (observability, CI/CD) cannot support a distributed system — the overhead will kill velocity rather than improve it.

Full answer: Microservices are a solution to specific problems — independent scaling of different components, team autonomy, and polyglot technology choices — not a default architectural pattern. The case for microservices is strong when: different components have wildly different scaling requirements (search needs GPU-backed instances, checkout needs CPU); multiple large teams need to deploy independently without coordination; and the domain is large enough that a single codebase becomes a bottleneck. The case against is equally strong. With a small team — fewer than five to eight engineers — the overhead of distributed systems (network calls instead of in-process calls, distributed tracing, per-service CI/CD, separate deployments) will destroy velocity more than the benefits justify. At an early-stage startup, you do not yet know where the domain boundaries are, so you will inevitably cut them wrong, creating a distributed monolith — the worst of both worlds. A distributed monolith is the anti-pattern where services must deploy together, share a database, or form synchronous call chains — you pay all the complexity costs of microservices with none of the independence benefits. Low operational maturity is another blocker: without Kubernetes or equivalent orchestration, centralized logging, distributed tracing, and per-service CI/CD pipelines, running 20 microservices becomes an operational nightmare. My recommendation: start with a well-structured modular monolith, establish clear module boundaries, and extract services only when a specific component’s scaling or team independence requirements genuinely justify the cost.

Name the distributed monolith anti-pattern explicitly — interviewers love when candidates can name the failure mode, not just describe the success pattern.

Gotcha follow-up: What is the strangler fig migration pattern, and how does it help you migrate from a monolith? The strangler fig pattern, named after the tree that grows around a host tree and gradually replaces it, is the safest approach to migrating from a monolith to microservices. You place a proxy or facade in front of the monolith and begin routing specific features — user authentication, for example — to a new microservice while all other traffic continues hitting the monolith. Over time you route more features to new services, shrinking the monolith incrementally until it can be decommissioned. During the transition, you use dual-write patterns or Change Data Capture to keep the monolith’s database and the microservice’s database in sync. The key principle: never do a big-bang rewrite. A big-bang rewrite means running two versions in parallel for months, reconciling diverging features, and accepting massive risk at the cutover. The strangler fig reduces risk by making the migration incremental and reversible at each step.

Q2: What is the difference between an API Gateway and a BFF? Concept Check

One-line answer: An API Gateway is infrastructure-level — generic routing, auth, SSL for all clients; a BFF is application-level — a purpose-built, client-specific aggregation layer owned by the frontend team.

Full answer: Both an API Gateway and a BFF sit at the edge of your system and intercept client requests, but they operate at different layers and serve different purposes. An API Gateway is infrastructure: it handles routing, SSL/TLS termination, JWT authentication, rate limiting, request transformation, and observability injection. It is generic — it serves all client types the same way — and is owned and managed by the platform or infrastructure team. It should contain no business logic. A BFF, which stands for Backend for Frontend, is application-level: it is a purpose-built service tailored to the needs of one specific client type. The web BFF might return rich, detailed JSON with all fields needed for a desktop dashboard. The mobile BFF returns compressed, minimal payloads suitable for slow cellular connections and small screens. The partner API BFF returns responses shaped to a B2B partner’s contract. Each BFF is owned by the frontend team for that client — they can change response shapes without coordinating with other teams. The BFF also aggregates data from multiple microservices into a single response optimized for the client’s view — something the infrastructure gateway must never do. In a large system you might have both: an API Gateway at the edge handling auth and SSL, with separate BFF services behind it, each serving their respective client type.

Use a concrete example: “The mobile BFF might combine data from three microservices and return a single compact response — the API Gateway just routes the request to the right BFF.”

Gotcha follow-up: What are the risks of the BFF pattern becoming a bottleneck? Each BFF is a network hop and a potential single point of failure. If the web BFF goes down, the entire web frontend loses its backend. Mitigations: deploy multiple BFF instances behind a load balancer; keep BFF logic thin — it should aggregate and transform, not implement complex business rules; use timeouts and circuit breakers on the BFF’s calls to downstream microservices so a slow microservice does not make the entire BFF unresponsive. The BFF also risks becoming a dumping ground for business logic over time — teams add convenient shortcuts. Enforce the rule that BFFs contain only presentation-layer logic: field selection, response shaping, and aggregation. Business rules belong in the domain microservices.

Q3: How does service discovery work in a microservices cluster? Concept Check

One-line answer: Service discovery solves the dynamic IP problem — instances start and stop constantly, so services use a registry to find each other rather than hardcoded addresses.

Full answer: In a microservices cluster on Kubernetes or a VM-based environment, service instances are created and destroyed dynamically — their IP addresses change whenever a pod restarts or scales. Hardcoding IP addresses in configuration files is not viable. Service discovery solves this. There are two models. Client-side discovery: each service instance registers itself with a service registry — Netflix Eureka is the classic example — publishing its IP, port, and health status. When Service A needs to call Service B, it queries the registry for a list of healthy Service B instances and then load-balances the request itself using a client-side load balancer like Netflix Ribbon. This gives the client full control over load balancing strategy (round-robin, weighted, zone-aware), but every language needs a registry client library, which is a significant operational burden in a polyglot environment. Server-side discovery: the client calls a stable DNS name or virtual IP address — a Kubernetes Service, for example. Kubernetes maintains a stable ClusterIP and DNS name for each Service resource, and kube-proxy or iptables routes the request to a healthy pod. The client knows nothing about individual instances — it just makes an HTTP call to a stable address, and the infrastructure handles routing. This is far simpler for clients and works transparently for any language or protocol. In Kubernetes, the combination of Service resources for stable addresses plus readiness probes for health filtering gives you complete service discovery out of the box.

In Kubernetes contexts, always mention that a Service resource provides a stable ClusterIP and DNS name — it is the most common implementation interviewers expect you to know.

Gotcha follow-up: What happens to in-flight requests during a pod restart in Kubernetes? When Kubernetes terminates a pod, it sends SIGTERM to the container’s process and simultaneously removes the pod from the Service’s Endpoints — stopping new traffic from being routed to it. However, kube-proxy updates are not instantaneous: there is a small window where the Service may still route requests to the terminating pod. To handle this gracefully: the application should handle SIGTERM by stopping acceptance of new connections and completing in-flight requests before shutting down — a graceful shutdown with a drain period. In Spring Boot, server.shutdown=graceful and spring.lifecycle.timeout-per-shutdown-phase=30s configure this. The terminationGracePeriodSeconds in the pod spec gives the container time to drain before Kubernetes force-kills it with SIGKILL.

Q4: Explain the Circuit Breaker pattern and its three states. Concept Check

One-line answer: A circuit breaker prevents cascading failure by fast-failing calls to a degraded downstream service, giving it recovery time, rather than allowing all threads to block waiting for a service that cannot respond.

Full answer: The circuit breaker pattern is named after the electrical circuit breaker that cuts power to protect equipment from overload. In microservices, the problem it solves is cascading failure: Service A calls Service B, Service B is slow or returning errors, and all of Service A’s threads block waiting for Service B. Eventually Service A’s thread pool exhausts and Service A starts failing too — a failure cascade that can take down the entire system. The circuit breaker has three states. CLOSED is the normal state: requests flow through to Service B, and failures are counted in a sliding window. When the failure rate exceeds a configured threshold — for example, 50% failure rate in the last 10 seconds — the circuit transitions to OPEN. OPEN is the protection state: all requests to Service B are failed immediately without making a network call — this is called fast-failing. Service A’s threads are not blocked; they fail instantly and can return a fallback response (cached data, a default value, or a degraded UI) instead of hanging. The circuit stays open for a configured wait duration — for example, 30 seconds — giving Service B time to recover. After the wait duration, the circuit transitions to HALF-OPEN: a configurable number of probe requests are allowed through to Service B. If those probes succeed, the circuit closes (back to normal). If they fail, the circuit opens again for another wait period. The key insight: without a circuit breaker, a slow downstream service can take down the entire call chain. With one, the impact is isolated and controlled.

Use the cascading failure story explicitly: “Without a circuit breaker, Service B’s slowness exhausts Service A’s thread pool and Service A starts failing — one service problem becomes a system-wide outage.”

Gotcha follow-up: What is the difference between a circuit breaker and a retry with exponential backoff? They solve different problems and should be used together. Retry with exponential backoff handles transient failures — brief network glitches or momentary unavailability — by attempting the call again with increasing delays between attempts. It assumes the failure is temporary and self-correcting within seconds. A circuit breaker handles sustained failures — a downstream service that is down or overloaded for minutes or longer. If you have a retry strategy without a circuit breaker, and Service B is down for 5 minutes, every incoming request to Service A will wait through the full retry sequence (3 retries at 1s, 2s, 4s = 7 seconds of blocking per request) for 5 minutes — hammering Service B with retry storms. The circuit breaker short-circuits this: once the failure rate threshold is hit, it stops all calls to Service B immediately, giving it breathing room to recover. The correct layering: @Retry (innermost) wrapped by @CircuitBreaker (outermost) — in Resilience4j this is the correct annotation order.

Q5: How do you apply Resilience4j annotations in Spring Boot? Concept Check

One-line answer: Resilience4j provides @CircuitBreaker, @Retry, @RateLimiter, @Bulkhead, and @TimeLimiter as method-level annotations in Spring; combine them in the right order and configure each instance in application.yml.

Full answer: Resilience4j is the successor to Netflix Hystrix for resilience patterns in Spring Boot. Each pattern is available as a method-level annotation. @CircuitBreaker wraps a method with a circuit breaker — configure the failure rate threshold, slow call threshold, and wait duration in the open state. Specify a fallbackMethod that takes the same parameters plus a Throwable — this method is called when the circuit is open or the call fails: @CircuitBreaker(name = “inventoryService”, fallbackMethod = “fallbackInventory”). @Retry retries the method on specified exceptions — configure maxAttempts and waitDuration with exponential backoff. @RateLimiter limits the rate of outgoing calls to a downstream service — useful for respecting a partner API’s rate limits. @Bulkhead isolates thread resources: SEMAPHORE type limits concurrent calls using the calling thread; THREADPOOL type uses a separate thread pool per downstream and returns CompletableFuture, providing true isolation. @TimeLimiter wraps an async call with a timeout — the method must return CompletableFuture for this to work. The correct stacking order when combining annotations: @CircuitBreaker wraps @Retry wraps @TimeLimiter. This means: attempt the call with a timeout; if it fails, retry; if enough retries fail, open the circuit. Configure each named instance in application.yml under resilience4j.circuitbreaker.instances.inventoryService — keeping configuration out of annotations enables environment-specific tuning without code changes.

Always mention the stacking order: TimeLimiter innermost, Retry middle, CircuitBreaker outermost — getting this wrong inverts the intended behavior.

Gotcha follow-up: What is the difference between Semaphore Bulkhead and ThreadPool Bulkhead? Semaphore Bulkhead limits the number of concurrent calls to a downstream service using a counting semaphore. Threads from the calling service’s thread pool enter the semaphore, make the call, and return. If the semaphore limit is reached, the call is rejected immediately. This is lightweight but does not prevent the calling service’s thread pool from being occupied — a slow downstream still holds threads from the main pool. ThreadPool Bulkhead uses a completely separate, bounded thread pool for calls to a specific downstream. The calling thread hands the work to the dedicated pool and gets back a CompletableFuture — the calling thread is immediately freed. This provides true isolation: even if the dedicated pool is exhausted by a slow downstream, the main thread pool is unaffected. ThreadPool Bulkhead is the stronger isolation guarantee; Semaphore is lower overhead and suitable when thread pool exhaustion is not a concern.

Q6: Compare the Saga choreography and orchestration patterns for distributed transactions. Tradeoff Question

One-line answer: Choreography uses event-driven reaction with no central coordinator — loosely coupled but hard to trace; orchestration uses a central Saga Orchestrator with explicit commands — visible flow but introduces coupling to the orchestrator.

Full answer: The Saga pattern exists because microservices with separate databases cannot use a single ACID transaction. If placing an order requires updating the Order Service, deducting from the Inventory Service, and charging the Payment Service, you cannot wrap these three operations in a database transaction — they span three databases. The Saga pattern chains local transactions, with each step either succeeding and triggering the next step, or failing and triggering compensating transactions to undo previous steps. In choreography, services react to events autonomously. Order Service places the order and publishes an OrderCreated event. Inventory Service listens, reserves inventory, and publishes InventoryReserved. Payment Service listens, charges the customer, and publishes PaymentCharged. If Payment fails, it publishes PaymentFailed, Inventory Service listens and releases the reservation, and Order Service listens and cancels the order. This is loosely coupled — services do not know about each other directly. The downside: there is no single place to see the entire business process. Debugging requires correlating events across multiple service logs. In orchestration, a central Saga Orchestrator sends explicit commands: “Reserve Inventory,” “Charge Payment,” and receives replies. The flow is fully visible in one place — you can draw a flowchart from the orchestrator’s code. The downside: the orchestrator has a dependency on every service in the saga, and it is a potential bottleneck. Tools that support orchestration: Temporal (durable workflow execution), Axon Framework (CQRS + Saga), and AWS Step Functions for serverless workloads.

Always distinguish the two failure paths: for choreography it is implicit (service reacts to failure events); for orchestration it is explicit (orchestrator decides to execute compensating commands).

Gotcha follow-up: What is the difference between a compensating transaction and a rollback? A database rollback is atomic — the database undoes the change as if it never happened, with no external visibility. A compensating transaction is a new, separate transaction that semantically undoes the effect of a previous committed transaction. For example, if the Order Service committed an order record and the Payment step later fails, the compensating transaction for the Order step is a new UPDATE setting order status to CANCELLED — it does not remove the original INSERT. This means compensating transactions are visible in the audit log, they can fail themselves (requiring retry logic), and they may not be able to fully reverse certain operations (you cannot un-send an email). Saga design must account for these irreversible actions — this is why payment services often place holds rather than immediately charging, allowing the hold to be released as the compensating action.

Q7: How does distributed tracing work, and what is the OpenTelemetry standard? Concept Check

One-line answer: Distributed tracing tracks a single request’s journey across all microservices using a shared trace ID propagated in HTTP headers; OpenTelemetry is the vendor-neutral SDK for emitting and exporting these traces.

Full answer: When a single user request flows through ten microservices, a log line in one service tells you almost nothing about why the request was slow or where it failed. Distributed tracing solves this by linking all related operations across services with a shared identifier. The key concepts: a Trace represents the end-to-end journey of one request — it has a unique trace ID. A Span represents one unit of work within the trace — one service handling part of the request, one database query, one HTTP call. Each span has a start time, end time, span ID, and parent span ID — the parent-child relationship forms a tree structure showing the call hierarchy. A span also carries tags (key-value metadata like http.method, db.statement) and events (timestamped log entries within the span). The W3C Trace Context standard defines how context is propagated between services: the traceparent HTTP header carries 00-{traceId}-{spanId}-{flags}. When Service A calls Service B, it passes this header, and Service B creates a child span using the received trace ID and parent span ID. OpenTelemetry is the vendor-neutral observability framework — a merger of OpenCensus and OpenTracing — that provides a single SDK for emitting metrics, logs, and traces regardless of which backend you use (Jaeger, Zipkin, Grafana Tempo, Datadog, AWS X-Ray). In Spring Boot 3, add spring-boot-starter-actuator, micrometer-tracing-bridge-otel, and an OpenTelemetry exporter to your dependencies — Spring auto-instruments all HTTP requests, Kafka consumers, and JDBC calls.

Connect the theory to Spring Boot 3 implementation — mention Micrometer Tracing + OpenTelemetry bridge to show production knowledge.

Gotcha follow-up: What is the difference between a trace, a span, and a log entry — and how are they related? A trace is the highest-level concept: the entire story of one request across all services, identified by a trace ID. A span is a chapter in that story: one service’s processing, one database query, one external HTTP call — a span has timing information and belongs to exactly one trace. A log entry is a text event that occurs at a point in time and can be correlated to a span by embedding the trace ID and span ID in the log message. The key relationship: a log entry by itself is a random line in a sea of logs. But when the trace ID is embedded in the log message, you can retrieve all log entries across all services for a specific user request in one query — this is called correlated logging. In Spring Boot, Micrometer Tracing automatically injects trace ID and span ID into the MDC (Mapped Diagnostic Context) so every log line emitted during request processing automatically includes the trace context.

Q8: What is the Bulkhead pattern, and what failure does it prevent? Concept Check

One-line answer: The Bulkhead pattern isolates resource pools (thread pools or connection counts) per downstream dependency so that a slow or failing dependency can only exhaust its own pool, not bring down the entire service.

Full answer: The bulkhead pattern is named after the watertight compartments in a ship’s hull that prevent flooding in one compartment from sinking the ship. The failure it prevents: resource pool exhaustion through a shared dependency. Consider Service A that calls both Service B and Service C using a single shared thread pool of 50 threads. Service B becomes slow — responses take 10 seconds each. Incoming requests to Service A that need Service B start consuming threads, waiting 10 seconds each. After a few seconds, all 50 threads are blocked waiting for Service B. Now a request arrives that only needs Service C — which is completely healthy — but there are no threads available to serve it. Service C’s functionality is unavailable not because Service C is broken, but because Service B exhausted the shared thread pool. The bulkhead pattern fixes this by giving each downstream dependency its own isolated resource pool. Service B calls use a pool of 10 threads; Service C calls use a separate pool of 10 threads. Service B can exhaust its own pool without affecting Service C’s pool at all. Resilience4j implements this with @Bulkhead: SEMAPHORE type limits concurrent calls using the calling thread, lightweight but does not fully isolate from the main thread pool. THREADPOOL type uses a completely separate bounded thread pool — the calling thread hands the work off and gets a CompletableFuture back immediately, providing true isolation. The trade-off: each isolated pool uses threads that could otherwise be shared, so you need to size your thread pools based on actual downstream concurrency requirements.

Open with the ship analogy — it is memorable and precisely describes the mechanism: isolated compartments prevent one failure from propagating.

Gotcha follow-up: How do you size a bulkhead thread pool in production? Bulkhead thread pool sizing follows Little’s Law: the optimal pool size equals the average throughput (requests per second) multiplied by the average latency (seconds) of the downstream call. For example, if Service B handles 100 calls per second with an average latency of 200ms (0.2 seconds), the steady-state concurrency is 100 × 0.2 = 20 concurrent requests — so a pool of 20 to 30 threads is appropriate. Size too small: requests are rejected even when the downstream is healthy, causing unnecessary failures. Size too large: the pool provides insufficient isolation because it can still consume significant resources before exhaustion. Monitor actual concurrency in production using metrics like resilience4j.bulkhead.available_concurrent_calls and adjust pool sizes accordingly.

Q9: What is a service mesh, and when is it worth the operational complexity? Tradeoff Question

One-line answer: A service mesh offloads cross-cutting service-to-service concerns (mTLS, retries, circuit breaking, tracing) into sidecar proxies managed by the infrastructure layer, removing that code from every application.

Full answer: A service mesh is an infrastructure layer that intercepts all network communication between microservices. In Istio, the most widely adopted service mesh, each pod gets an Envoy sidecar proxy automatically injected by Kubernetes admission webhooks. All inbound and outbound network traffic passes through the Envoy sidecar, not directly to the application. The control plane, called Istiod, distributes configuration to all Envoy sidecars centrally. What this enables, without any application code changes: automatic mutual TLS (mTLS) between all services — each sidecar validates the other’s certificate, providing zero-trust network security between services; traffic management — canary deployments (route 10% of traffic to v2), blue-green deployments, circuit breaking, retries, and timeouts configured declaratively in YAML; distributed tracing — Envoy emits spans for every service-to-service call automatically; and detailed traffic metrics for every service pair. The trade-offs are significant: each Envoy sidecar adds approximately 1ms of latency per hop and consumes 50-100MB of RAM per pod. Istiod and its components add operational complexity and another system to learn and maintain. The control plane itself must be highly available. A service mesh is justified at significant scale — 50+ services, strict security requirements (PCI-DSS, HIPAA), or when the operational cost of implementing these concerns in application code exceeds the mesh overhead. For a small cluster with 10 services, Resilience4j in the application is simpler and sufficient.

Be balanced: mention both the power (zero-code mTLS, canary deployments) and the real costs (latency, RAM, operational complexity) — interviewers respect candid trade-off analysis.

Gotcha follow-up: If the service mesh handles retries and circuit breaking, should you also implement them in the application with Resilience4j? There is a risk of double-retrying: if both the service mesh and the application retry a failing request, the downstream service receives exponentially more load. For example, if the application retries 3 times and the mesh retries 3 times, a single application call generates up to 9 requests to the downstream. The correct approach is to choose one layer: use either the mesh or the application library, not both. In a service mesh environment, I would configure retries and circuit breaking at the mesh layer and remove them from application code — keeping the application code focused on business logic. Exception: application-level bulkheads (thread pool isolation) are still valuable even with a mesh, because the mesh operates at the network layer and cannot prevent thread pool exhaustion in the application.

Q10: How does the Strangler Fig pattern work for migrating a monolith? Design Scenario

One-line answer: You route a growing subset of features from the monolith to new microservices via a proxy, incrementally replacing functionality until the monolith can be decommissioned — never doing a big-bang rewrite.

Full answer: The strangler fig tree grows around a host tree, eventually replacing it entirely while the original tree slowly disappears. The pattern translates directly to monolith migration. The first step is to place a routing proxy — an API Gateway or Nginx — in front of the monolith. All traffic still flows through the proxy to the monolith initially. The migration proceeds feature by feature: identify a bounded context that is relatively independent — user authentication is a common first choice. Extract it into a new microservice, move its data to a new database, and update the proxy to route authentication requests to the new service. The monolith still handles everything else. Over multiple sprints, route more features to new services, shrinking the monolith’s responsibilities with each iteration. The data synchronization challenge is the hardest part: during the transition, the monolith’s database and the new microservice’s database may both need to hold the same data. Solutions include dual-write (write to both databases on every update, comparing results), or Change Data Capture using a tool like Debezium — which reads the monolith’s database transaction log and streams changes to the new service’s database in near-real-time. The key principle: every step is independently deployable and reversible. If the new user authentication service has a bug, you flip the proxy routing back to the monolith — the migration is not a one-way door at any point. This is the fundamental advantage over a big-bang rewrite, which requires running two codebases in parallel for months and accepting massive risk at a single cutover moment.

Always mention Change Data Capture (CDC) and Debezium — the data sync problem is the most technically complex part of the pattern and shows depth.

Gotcha follow-up: How do you handle the period when both the monolith and the new service write to different databases that must stay in sync? The dual-write approach writes to both databases synchronously on every operation. This is simple but has a consistency risk: if the write to the new service’s database fails after the monolith’s write succeeds, you have diverged state. A more robust approach is the Outbox Pattern: the monolith writes its business event to an Outbox table in the same local database transaction as its business data change — guaranteeing atomicity. A background process (or Debezium CDC) reads the Outbox table and reliably delivers the event to the new service’s database or a message broker. This guarantees that every monolith change is eventually reflected in the new service’s data store, even in the face of failures, without two-phase commit. Once you are confident the new service’s data is consistent, you can stop writing to the monolith’s copy of that data.

Q11: What does “database per service” mean, and how do you handle queries that span multiple services? Concept Check

One-line answer: Each microservice owns its data exclusively — no direct cross-service database access — enabling independent evolution and polyglot persistence, with cross-service queries handled via API Composition, CQRS, or materialized event-driven views.

Full answer: The database-per-service principle is one of the most important microservices tenets. Each service has its own database (or at minimum its own schema) that no other service is allowed to access directly — no shared tables, no cross-service JOINs. The benefits: the Order Service can change its schema without coordinating with the Customer Service. Each service can use the storage technology best suited to its access patterns — PostgreSQL for transactional order data, Cassandra for time-series events, Elasticsearch for full-text search, Redis for session data. And each service can scale its database independently. The challenge: you lose the ability to do a single SQL JOIN across services. If you need to display an order with the customer’s name and the product details, you cannot join orders, customers, and products in one query. The solutions: API Composition — the API layer (BFF or API Gateway) calls the Order Service, Customer Service, and Product Service in parallel and merges the results in memory. This is simple but requires multiple network calls. CQRS with event-driven views — services publish events to Kafka when data changes; a read model service subscribes to these events and builds a denormalized, pre-joined view optimized for queries. The read model is eventually consistent but requires zero network calls at query time. Distributed transactions across services use the Saga pattern — never a two-phase commit, which creates tight coupling and is a distributed systems anti-pattern.

Name all three cross-service query strategies: API Composition (simplest), CQRS read models (highest performance), and Saga for transactions.

Gotcha follow-up: What is polyglot persistence, and what operational problem does it create? Polyglot persistence means using different database technologies for different services based on their access patterns — PostgreSQL for relational transactional data, Cassandra for high-write time-series data, Elasticsearch for full-text search, Neo4j for graph traversals, Redis for caching and sessions. The benefit is that each service uses the optimal tool for its workload rather than forcing every service onto one general-purpose database. The operational problem: your team must now understand, operate, monitor, backup, and tune multiple different database systems. Cassandra has different operational characteristics from PostgreSQL — different failure modes, different consistency guarantees, different operational tooling. Each database type requires specialized expertise. At scale, with a large platform team, this is manageable. At smaller scale, the cognitive overhead of operating many different database technologies can outweigh the performance benefits. My approach: start with PostgreSQL for all services — it handles most workloads well — and introduce a specialist database only when a service’s requirements cannot be met by PostgreSQL.

Q12: How should JWT tokens be propagated and validated across microservices? Design Scenario

One-line answer: The API Gateway validates the JWT once; downstream services either re-validate it using the authorization server’s public key (stateless, no network call) or trust enriched headers the gateway appended after validation.

Full answer: JWT propagation requires a clear security architecture decision. In the first approach — direct JWT propagation — the API Gateway validates the JWT signature and expiry using the Authorization Server’s public key. If valid, it forwards the original Authorization: Bearer header to all downstream services. Each downstream service independently re-validates the JWT using the public key (RS256 asymmetric signing means any service can validate with the public key, no network call required) and extracts claims like user ID, email, and roles. This is stateless, horizontally scalable, and requires no coordination. The downside: every microservice needs JWT validation libraries and access to the public key or JWKS (JSON Web Key Set) endpoint. In the second approach — gateway-enriched headers — the gateway validates the JWT once and then forwards the user context as simple HTTP headers: X-User-Id: 12345, X-User-Roles: admin,read. Downstream services trust these headers without doing JWT validation themselves — they just read the header values. This approach keeps JWT logic isolated to the gateway, but it assumes all traffic genuinely comes through the gateway. If any service is accidentally exposed directly — bypassing the gateway — there is no authentication check. The hybrid approach: validate JWT at the gateway (fail early), forward the original JWT downstream, and have services validate it independently for defense in depth. For sensitive operations that require fresh authorization, services should call the Authorization Server’s introspection endpoint to check if the token has been revoked — accepting that this is a network call with caching implications.

Frame the trade-off clearly: gateway-enriched headers are simpler but require strict network isolation; direct JWT forwarding is defense-in-depth but adds JWT library dependencies.

Gotcha follow-up: How do you handle JWT token revocation in a stateless system? JWTs are stateless — once issued, they are valid until they expire. Revoking a JWT (for example, after a user logs out or their account is suspended) is fundamentally at odds with statelessness. Solutions in order of complexity: short expiry (5-15 minutes) plus refresh tokens — the access JWT expires quickly, limiting the revocation window; a token revocation list (blocklist) in Redis — on logout, store the token’s JTI (JWT ID) in Redis with a TTL equal to the token’s remaining lifetime; on each request, services check the blocklist. This re-introduces state but only for revoked tokens. A third option is to call the Authorization Server’s token introspection endpoint on every request — fully accurate but a network call on every request, introducing latency and a dependency. The production standard for most systems: short-lived access tokens (15 minutes) plus a Redis blocklist for immediate revocation on logout or account suspension.

Q13: What are Kubernetes liveness, readiness, and startup probes, and when do you use each? Concept Check

One-line answer: Liveness detects stuck processes and triggers restart; readiness detects unready dependencies and removes the pod from load balancing; startup provides a grace period for slow-starting apps before liveness kicks in.

Full answer: Kubernetes uses three probe types to manage pod lifecycle. A liveness probe answers the question: “Is this process alive and not stuck in a fatal state?” Kubernetes calls the liveness probe endpoint on a configurable schedule. If it fails a configurable number of times consecutively, Kubernetes restarts the container. The liveness probe should check internal application state: not in a deadlock, not out of memory, not stuck in an infinite loop. It must not check external dependencies — if the database is down, that is a readiness issue, not a liveness issue. Restarting the application does not fix a database outage. Spring Boot Actuator exposes /actuator/health/liveness for this purpose. A readiness probe answers: “Is this service ready to accept traffic?” If it fails, Kubernetes removes the pod from the Service’s Endpoints — no new traffic is routed to it — but the container is not restarted. This is the correct response to dependency failures: the database is down, so stop sending traffic to this pod until it recovers. Spring Boot: /actuator/health/readiness. A startup probe handles applications that take a long time to start — Spring Boot with many auto-configurations, applications that pre-warm caches on startup. Without a startup probe, the liveness probe might start failing immediately after container start (because the app is still initializing) and restart the container in a loop before it ever finishes starting. The startup probe gives the application an initial grace period — a longer timeout — before the liveness probe begins. Once the startup probe succeeds, Kubernetes switches to the regular liveness probe.

Always explain WHY liveness must not check external dependencies — this is the most common misconfiguration interviewers probe for.

Gotcha follow-up: What happens if your liveness probe calls an external database and the database goes down? All pods of that service will fail their liveness probe and be restarted by Kubernetes in a loop — a Crash Loop Backoff — even though the application code is perfectly healthy. Restarting the application does not fix the database outage. You have turned a recoverable dependency failure into a service-wide restart storm. The correct fix: liveness probes check only internal application state (heap health, deadlock detection, internal queue health). External dependency health — database connectivity, downstream service reachability — belongs in the readiness probe. When the database comes back, the readiness probe starts passing, pods are added back to the load balancer, and traffic resumes — no restarts needed.

Q14: What is 12-Factor App config externalization, and how do you implement it in Spring Boot? Concept Check

One-line answer: Factor III of the 12-Factor App says config that varies between environments must never be in code — store it in environment variables, Kubernetes ConfigMaps/Secrets, or Spring Cloud Config Server with @RefreshScope for live reload.

Full answer: The 12-Factor App methodology defines Factor III: Config. Config is anything that varies between deployment environments — database URLs, API keys, feature flags, service endpoints. If you hardcode these values or commit them to version control, deploying to a new environment requires a code change — which breaks the principle that the same artifact deploys to any environment. The correct approach: inject all environment-specific config at runtime. In Kubernetes, ConfigMaps hold non-sensitive config and Secrets hold sensitive values (passwords, API keys) — both are mounted as environment variables or files into the container. Spring Boot reads environment variables automatically and maps them to @ConfigurationProperties using relaxed binding (SPRING_DATASOURCE_URL becomes spring.datasource.url). Spring Cloud Config Server provides a centralized Git-backed config store: services fetch their config from the Config Server at startup, and with @RefreshScope on beans, can reload config dynamically without a restart when the Config Server sends a refresh event or Spring Cloud Bus broadcasts it. For secrets specifically: Kubernetes Secrets are Base64-encoded, not encrypted — for production security, use AWS Secrets Manager, HashiCorp Vault, or Azure Key Vault with a Spring integration that fetches secrets at startup and rotates them without restarts. The benefit of full externalization: the same Docker image (JAR) can run in dev, staging, and production — only the environment’s config differs. This is the foundation of continuous delivery.

Mention @RefreshScope specifically — it demonstrates knowledge of dynamic config reload, which is a production-grade concern most candidates skip.

Gotcha follow-up: What is the risk of using environment variables for secrets in Kubernetes? Environment variables in Kubernetes are not encrypted at rest by default — Kubernetes Secrets are only Base64-encoded, which is encoding, not encryption. Environment variables are also visible in pod describe output, accessible to anyone with kubectl exec or kubectl describe pod permissions. Mitigations: enable etcd encryption at rest for Secrets in Kubernetes. Use a dedicated secrets manager (AWS Secrets Manager, HashiCorp Vault) with the Vault Agent Sidecar or External Secrets Operator — these fetch secrets from the vault at pod startup and inject them as environment variables or mounted files, ensuring the actual secret value is never stored in Kubernetes etcd. Implement RBAC to restrict who can kubectl exec into pods or read Secrets. Rotate secrets regularly and audit access logs.

Q15: What are the 12-Factor App principles most relevant to backend interviews? Concept Check

One-line answer: The factors most cited in interviews are: Config in environment (III), Stateless processes (VI), Disposability (IX), Dev/prod parity (X), and Logs as streams (XI) — each maps directly to a cloud-native scalability or operational requirement.

Full answer: The 12-Factor App, published by Heroku in 2012, is a methodology for building applications that are portable, scalable, and maintainable in cloud environments. In backend interviews, five factors come up repeatedly. Factor II (Codebase): one codebase tracked in version control, many deploys — no separate codebases per environment, no branching per environment. Factor III (Config): everything that varies between deployments lives in environment variables, never in code. Factor VI (Processes): application processes are stateless and share nothing — session state lives in Redis, not in-memory. This is the prerequisite for horizontal scaling: if a process holds session state, you cannot add more instances because clients must always reach the same instance. Factor IX (Disposability): fast startup — under 30 seconds — and graceful shutdown — handle SIGTERM, finish in-flight requests, release resources cleanly. Fast startup enables scaling; graceful shutdown prevents request drops during deployments. In Kubernetes: server.shutdown=graceful in Spring Boot. Factor X (Dev/prod parity): minimize the gap between development and production environments. Use Docker Compose locally to run the same PostgreSQL version as production, the same Kafka version, the same Redis version. Differences between dev and prod are where “works on my machine” bugs live. Factor XI (Logs as streams): write logs to stdout/stderr, never to files. The platform (Kubernetes, ECS) captures stdout and routes it to your logging system (ELK, Grafana Loki). Applications must not manage log files, log rotation, or log shipping.

Frame your answer as “the factors that directly constrain architecture decisions” — this shows you understand the principles, not just the list.

Gotcha follow-up: How does Factor VI (stateless processes) affect session management? If application processes are stateless, HTTP sessions cannot be stored in-memory in the JVM — because subsequent requests from the same user may hit a different server instance. The solution is external session storage: Spring Session provides transparent session storage in Redis. The JSession cookie still works from the browser’s perspective, but the session data is read from and written to Redis on every request. Any server instance can handle any request because they all share the same session store. This also means session data survives server restarts — a user’s session persists across deployments. The trade-off: Redis becomes a dependency, and its availability directly affects the ability to authenticate users. Mitigation: Redis cluster with replication, and graceful degradation if Redis is temporarily unavailable (redirect to login rather than a 500 error).

Common Mistakes:

  • Building a distributed monolith → services that share a database or deploy together give you microservices complexity with none of the independence. Define clear domain ownership before splitting.
  • Liveness probe checking external dependencies → if the DB is down, Kubernetes restarts all pods in a loop. Liveness must check only internal process health; readiness checks dependencies.
  • Eager rebalancing in Kafka consumers (Section 3 preview) → also a microservices pattern: always use CooperativeStickyAssignor to prevent “stop the world” rebalances.
  • Committing offsets before processing (also Section 3 preview) → use manual acknowledgment after processing, not auto-commit, to avoid data loss.
  • Retries at both the service mesh and application layer → double-retrying multiplies load on the failing service. Choose one retry layer.
  • Ignoring graceful shutdown → without SIGTERM handling and drain periods, in-flight requests are dropped on every deployment. Configure server.shutdown=graceful.

Quick Revision: Database-per-service = no cross-service JOINs + Saga for transactions; Circuit Breaker = CLOSED → OPEN → HALF-OPEN; Stateless processes = session in Redis; Liveness ≠ readiness; BFF = client-specific aggregation layer.

Section 3: Kafka

Reading guide: focus on the relationship between producer settings (acks, idempotence, transactions), consumer settings (offset commit, isolation level), and delivery guarantees — interviewers probe how the settings connect to outcomes.

Q1: What are Kafka’s core concepts — topic, partition, offset, consumer group? Concept Check

One-line answer: A topic is a named channel, partitions are its ordered sub-logs and the unit of parallelism, offsets are the position of each message, and consumer groups allow parallel consumption where each partition is consumed by exactly one member.

Full answer: Understanding Kafka’s architecture requires building up from the primitives. A Topic is a named, logical channel for a stream of messages — analogous to a database table name. A topic is divided into Partitions: each partition is an ordered, immutable, append-only log. Messages within a partition are strictly ordered, but there is no ordering guarantee across partitions. Partitions are the fundamental unit of parallelism — more partitions means more consumers can process messages in parallel. An Offset is the position of a message within a partition: a monotonically increasing integer starting at 0. Messages are identified by the combination of topic + partition + offset. Offsets are never reused — they only increase. The Broker is a Kafka server process. A production Kafka cluster has multiple brokers for fault tolerance. For each partition, one broker is the Leader — it handles all reads and writes. Other brokers hold Followers — replicated copies of the partition that take over if the leader fails. A Consumer Group is a group of consumer instances that jointly consume a topic. Kafka assigns partitions to consumers such that each partition is consumed by exactly one consumer instance in the group at a time. This enables parallel processing: if a topic has 6 partitions and a consumer group has 3 instances, each instance gets 2 partitions. If you add a 7th partition, one instance gets 3. If you add a 7th consumer to a 6-partition topic, that consumer gets no partitions — you cannot have more consumers in a group than partitions.

Always mention the constraint: more consumers than partitions means idle consumers — this is a practical capacity planning point.

Gotcha follow-up: How does Kafka guarantee message ordering? Kafka guarantees ordering only within a single partition. All messages with the same key are routed to the same partition by the default partitioner (hash of the key modulo number of partitions), ensuring all messages for a given entity — all events for order ID 12345, for example — are in order within their partition. Across partitions, there is no global ordering guarantee. If you need global ordering across all messages in a topic, you must use a single partition — but this eliminates parallelism and limits throughput to what one partition can handle (typically hundreds of MB/second). For most use cases, per-key ordering (same key = same partition = ordered for that key) is sufficient and is the pattern you should design for.

Q2: What are the acks settings on a Kafka producer, and what are their reliability trade-offs? Tradeoff Question

One-line answer: acks=0 is fire-and-forget (fastest, message loss possible); acks=1 acknowledges the leader write (message loss if leader crashes before replication); acks=all acknowledges all ISR replicas (no data loss, slowest) — production always uses acks=all.

Full answer: The acks configuration controls how many broker acknowledgments the producer waits for before considering a message successfully sent — directly trading durability for latency and throughput. acks=0, called fire-and-forget, means the producer sends the message and moves on without waiting for any acknowledgment. This is the absolute maximum throughput mode, but if the broker crashes between receiving the message and writing it to disk, the message is silently lost. Use only for data where occasional loss is acceptable — metrics, access logs. acks=1 means the partition leader broker writes the message to its local log and sends an acknowledgment to the producer. If the leader crashes before any follower has replicated the message, the message is lost — the new leader elected from followers does not have it. Better than acks=0 but still not durable in all failure scenarios. acks=all (equivalent to acks=-1) means all In-Sync Replicas — the ISR, which is the set of replicas that are currently caught up with the leader — must write the message before the leader acknowledges it. This guarantees no data loss as long as at least one ISR replica is alive. This is the slowest option because the producer waits for all ISR brokers to confirm. The production recommendation is always: acks=all combined with min.insync.replicas=2 (so at least 2 replicas must confirm) and replication.factor=3 (3 total copies of each partition). This configuration tolerates one broker failure with no data loss.

Always pair acks=all with min.insync.replicas=2 — one without the other is incomplete.

Gotcha follow-up: What happens when min.insync.replicas=2 but only one replica is in the ISR? The producer receives a NotEnoughReplicasException — the write is rejected. The partition effectively becomes read-only: existing messages can still be consumed, but no new messages can be produced to it. This is an intentional design decision: Kafka chooses consistency over availability in this scenario. Rather than accept a write that cannot meet the minimum durability guarantee, it refuses the write. The operational implication: you must monitor ISR shrinkage with the kafka.server:type=ReplicaManager,name=IsrShrinksPerSec metric and alert when any partition’s ISR falls below min.insync.replicas. A broker being slow or lagging behind can cause it to be removed from ISR, reducing your durability headroom. Fix by investigating broker disk I/O, GC pressure, or network issues causing replication lag.

Q3: What is an idempotent producer, and what problem does it solve? Concept Check

One-line answer: The idempotent producer prevents duplicate messages caused by producer retries by having the broker deduplicate based on a producer ID and sequence number, guaranteeing exactly-once delivery per producer session.

Full answer: Without idempotent producers, a common failure scenario creates duplicates. The producer sends a message, the broker writes it to the partition log, and then the network drops before the broker’s acknowledgment reaches the producer. The producer sees a timeout, assumes the send failed, and retries — sending the same message again. The broker writes it a second time. The consumer now sees the message twice. This is the at-least-once problem for producers. The idempotent producer solves it. Enabled with enable.idempotence=true (default in Kafka 3.0+), the broker assigns each producer instance a unique Producer ID (PID) when it initializes. The producer also maintains a monotonically increasing Sequence Number per partition. Every message is tagged with both the PID and the sequence number. When the broker receives a message, it checks if it has already seen this PID + sequence number combination. If yes, it acknowledges the message without writing it again — deduplication at the broker level. The producer can safely retry any message on network failure with no risk of duplicates. Enabling idempotence automatically sets acks=all and caps max.in.flight.requests.per.connection to 5 to ensure the sequencing guarantees hold. The critical limitation: the PID is assigned per producer session. If the producer restarts (a new process, a pod restart), it gets a new PID. The broker cannot correlate messages across different PIDs, so a crash and restart can still cause duplicates at the boundary. For cross-restart deduplication, you need Kafka transactions.

Explicitly state the session scope limitation — this is the question interviewers follow up with and many candidates miss.

Gotcha follow-up: If idempotent producers prevent duplicates, why do we also need Kafka transactions? Idempotent producers prevent duplicate delivery of a single message within one producer session. Kafka transactions provide atomicity across multiple operations — either all of them commit or none do. The canonical use case is a consume-transform-produce pipeline: read messages from one topic, transform them, and produce results to another topic while committing the consumer offsets. With transactions, you wrap the produce calls AND the consumer offset commits in a single atomic transaction. Either the output messages are written AND the offsets are committed (the “done” state), or neither happens (the “retry from the same input” state). Without transactions, a crash between writing the output and committing the offsets would cause the consumer to re-process input messages and produce duplicate output — even with an idempotent producer. Transactions extend the exactly-once guarantee to the entire consume-transform-produce cycle.

Q4: What does exactly-once semantics in Kafka require, and what is the performance cost? Concept Check

One-line answer: Exactly-once requires transactional producers (transactional.id + transaction API) on the write side and isolation.level=read_committed on the read side — at roughly 20% throughput reduction.

Full answer: Exactly-once semantics (EOS) in Kafka means a message is produced exactly once and consumed and processed exactly once — no duplicates, no loss. This requires configuration on both the producer and consumer sides. On the producer side: set transactional.id to a stable, unique identifier for this producer instance (for example, “payment-service-partition-0”) — this ID must survive producer restarts to maintain transactional continuity. Call producer.initTransactions() on startup, then wrap your produce calls in producer.beginTransaction() / producer.commitTransaction() (or producer.abortTransaction() on failure). You can include consumer offset commits within the same transaction by calling producer.sendOffsetsToTransaction() — this atomically commits offsets and produced messages together. On the consumer side: set isolation.level=read_committed. Without this, the consumer reads all messages including those from aborted transactions and the transaction control markers (BEGIN, COMMIT, ABORT). With read_committed, the consumer only sees messages from committed transactions — it waits for the transaction to either commit or abort before returning messages from that transaction to the application. The consumer never sees messages from aborted transactions. The performance cost: transactions add coordination overhead — the producer must communicate with the Transaction Coordinator broker for beginTransaction and commitTransaction. Typical throughput reduction is approximately 20%. This cost is justified for financial transactions, order processing, inventory updates — any domain where duplicates or lost messages cause business errors.

Always pair the producer-side configuration with the consumer-side isolation.level=read_committed — candidates often describe only the producer half.

Gotcha follow-up: What is the zombie fencing problem in Kafka transactions, and how does transactional.id solve it? Zombie fencing is the problem of an old producer instance becoming active after it was presumed dead. Scenario: Producer A with transactional ID “payment-service-0” starts a transaction, becomes slow or paused (a GC pause, a network partition), and Kafka elects Producer B with the same transactional ID as the new active producer for that ID. Producer B starts a new transaction. Then Producer A “wakes up” and tries to commit its old transaction. Without fencing, A and B could both write to the same output topic and confuse the downstream. Kafka prevents this using an epoch counter. The Transaction Coordinator tracks the epoch for each transactional ID. When a new producer initializes with the same transactional ID, the coordinator increments the epoch. Any request from an older epoch — the zombie producer — is rejected with a ProducerFencedException. The zombie is fenced out by the epoch increment, ensuring only the most recently initialized producer for a given transactional ID can commit transactions.

Q5: What are the three message delivery semantics in Kafka? Concept Check

One-line answer: At-most-once commits offsets before processing (possible data loss, no duplicates); at-least-once commits after processing (no loss, possible duplicates); exactly-once uses transactions to guarantee neither loss nor duplicates.

Full answer: The three delivery semantics represent different trade-offs between data loss risk and duplicate processing risk. At-most-once: the consumer commits the offset immediately upon receiving the message, before doing any processing. If the consumer then crashes while processing the message, the message is never processed — its offset is already committed, so it will not be redelivered. Outcome: no duplicates, but possible data loss. Use for telemetry, metrics, or any data where occasional loss is acceptable. At-least-once: the consumer processes the message fully and commits the offset only after successful processing. If the consumer crashes after processing but before committing the offset, the message will be redelivered on restart — processed twice. Outcome: no data loss, but possible duplicates. This is the most common production approach. The prerequisite: make your consumer logic idempotent so processing the same message twice has the same effect as processing it once — use a unique message ID to detect and skip already-processed messages. Exactly-once: use Kafka transactions on the producer side (transactional.id, transaction API) and isolation.level=read_committed on the consumer side. The consumer offset commit and the produce of any output messages are wrapped in a single atomic transaction. Outcome: no loss, no duplicates. Most expensive — approximately 20% throughput reduction. Use for financial data, order processing, inventory management, any domain where duplicate or lost messages cause real business errors.

Frame the answer as a spectrum: “The more reliable the semantic, the higher the latency and complexity cost — choose based on business tolerance for loss vs. duplicates.”

Gotcha follow-up: How do you make a Kafka consumer idempotent to handle at-least-once duplicates? Idempotent consumer design depends on the processing type. For database writes: include a unique idempotency key in the message payload — for example, orderId or a UUID generated by the producer. Use an INSERT … ON CONFLICT DO NOTHING (PostgreSQL) or an upsert operation so a duplicate message does not create a duplicate database record. For external API calls: use the idempotency key pattern on the downstream API. For stateful aggregations: use a deduplication window — maintain a set of recently seen message IDs in Redis with a TTL, and skip processing for any ID that is already in the set. The window TTL must be longer than the maximum possible redelivery delay. For Kafka Streams: the framework handles deduplication internally when using exactly-once mode. The key insight: idempotency is a property you design into your processing logic — it does not come automatically from Kafka.

Q6: What is Kafka consumer rebalancing, and why should you use cooperative rebalancing? Concept Check

One-line answer: Rebalancing redistributes partitions when consumers join or leave a group; eager rebalancing (the old default) causes a “stop the world” pause; cooperative rebalancing (Kafka 2.4+) moves only the necessary partitions without stopping other consumers.

Full answer: A consumer rebalance occurs whenever the group membership changes — a consumer starts (joining the group), a consumer crashes or shuts down (leaving the group), or partitions are added to a topic. During a rebalance, Kafka must redistribute all partitions among the current group members. In eager rebalancing — the original mode using RangeAssignor or RoundRobinAssignor — all consumers revoke all their partitions simultaneously, stop consuming entirely, and wait for a new partition assignment from the group coordinator. This “stop the world” pause can last seconds to tens of seconds depending on group size. During this pause, all partitions accumulate lag — messages pile up unprocessed. In Kafka 2.4+, cooperative incremental rebalancing (CooperativeStickyAssignor) changes this behavior fundamentally. Only the partitions that need to move between consumers are revoked. All other consumers keep their existing partition assignments and continue consuming without interruption. The rebalance happens in two rounds: the coordinator first collects current assignments, computes the new assignment, identifies only the partitions that need to change hands, and instructs only those consumers to revoke and reassign. The improvement: a consumer group of 10 instances adding one new instance only briefly pauses the 1-2 consumers whose partitions are being moved, not all 10. To enable in Spring Kafka: add partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor to your consumer properties. This should be the default in any production consumer group.

Always mention that the default assignors use eager rebalancing — cooperative requires explicit configuration. This is a real production footgun.

Gotcha follow-up: What causes excessive consumer rebalances in production, and how do you prevent them? The most common cause is the consumer’s session.timeout.ms or heartbeat failures. The consumer sends heartbeats to the group coordinator on a separate background thread every heartbeat.interval.ms (default 3 seconds). If the coordinator does not receive a heartbeat within session.timeout.ms (default 10 seconds in older versions, 45 seconds in newer), it considers the consumer dead and triggers a rebalance. This can happen if: the consumer’s poll loop is processing a message that takes longer than max.poll.interval.ms (default 5 minutes) — increase this if your message processing is legitimately slow, or move processing to a separate thread so the poll thread stays responsive; the consumer machine is under heavy GC pressure — GC pauses can block the heartbeat thread; or network instability interrupts heartbeats. Solutions: tune max.poll.interval.ms to match your worst-case processing time; reduce max.poll.records so each poll batch is smaller and faster to process; use cooperative rebalancing so rebalances are less disruptive when they do occur; monitor rebalance frequency with the kafka_consumer_rebalance_total metric.

Q7: What is the difference between auto-commit and manual offset commit in Kafka, and which should you use? Tradeoff Question

One-line answer: Auto-commit commits offsets on a timer and can cause data loss if the consumer crashes between the commit and finishing processing; manual commit commits only after successful processing, guaranteeing at-least-once delivery.

Full answer: Offset management controls when Kafka considers a message “consumed” — it is the mechanism that determines what happens to a message if the consumer crashes. With enable.auto.commit=true — the default — Kafka commits offsets in the background every auto.commit.interval.ms milliseconds (default 5 seconds). This means the offset is committed based on time, not on whether your code has finished processing those messages. The failure scenario: the consumer polls a batch of 100 messages, auto-commit fires after 5 seconds and commits offset 100, and then the consumer crashes while processing message 80. Messages 80-100 were acknowledged to Kafka as consumed but were never actually processed — they are silently lost. This is at-most-once behavior. With manual commit — disable auto-commit by setting enable.auto.commit=false in Spring and using AckMode.MANUAL_IMMEDIATE — you call acknowledgment.acknowledge() explicitly inside your @KafkaListener method, but only after the message has been fully and successfully processed. If the consumer crashes before calling acknowledge(), the offset is not committed, and on restart the message is redelivered — at-least-once behavior. The trade-off: manual commit requires more code and careful exception handling. If your processing throws an exception and you never acknowledge, the consumer will repeatedly attempt to process the same message — a poison pill scenario. For this, you need a Dead Letter Topic (DLT): send the message to the DLT for out-of-band investigation, then acknowledge the original so the consumer can proceed.

Frame it as: “Auto-commit is convenient but gives at-most-once semantics — for any data where loss is unacceptable, always use manual commit.”

Gotcha follow-up: What is the “poison pill” problem in Kafka, and how do you handle it? A poison pill is a message that causes the consumer to throw a non-retryable exception every time it is processed — a malformed JSON payload, a message that violates a business invariant, or a message that causes an NPE in the processing code. Without DLT handling, the consumer cannot acknowledge this message, keeps retrying it, and makes no progress — the partition is stuck. Spring Kafka’s DefaultErrorHandler (formerly SeekToCurrentErrorHandler) handles this with backoff retry: on each failure it seeks back to the failed offset and retries a configurable number of times. After maxAttempts are exhausted, the message is forwarded to the Dead Letter Topic — typically named {original-topic}.DLT — and the original message is acknowledged, allowing the consumer to proceed. The DLT message includes headers with the original topic, partition, offset, and the exception stack trace for debugging. A separate process or periodic job should consume the DLT, investigate each message, fix the root cause, and potentially republish corrected messages to the original topic.

Q8: How do you tune Kafka producer throughput? Concept Check

One-line answer: The three key knobs are linger.ms (batch accumulation time), batch.size (max bytes per batch), and compression.type — together they trade a small latency increase for dramatically higher throughput by sending fewer, larger batches.

Full answer: Kafka’s producer throughput is largely determined by how efficiently messages are batched. By default, linger.ms=0 means the producer sends each message in its own network request the moment it is ready — one request per message, maximum latency, minimum throughput. Setting linger.ms to 5-20 milliseconds tells the producer to wait before sending, accumulating messages from the application into a batch. During that window, if other messages arrive for the same partition, they are bundled into a single network request — dramatically reducing per-message overhead. batch.size (default 16384 bytes = 16KB) controls the maximum bytes per batch per partition. When a batch reaches this size, it is sent immediately regardless of linger.ms. For high throughput, increase batch.size to 32KB or 64KB to allow larger batches. compression.type controls message compression per batch: none (no compression, highest throughput for CPU-bound systems), snappy (fast compression and decompression, good balance), lz4 (fastest decompression, good for consumer throughput), gzip (highest compression ratio, most CPU-intensive). For throughput: use lz4 or snappy. For storage efficiency: use gzip. buffer.memory (default 32MB) is the total amount of memory the producer uses for buffering messages waiting to be sent — increase this if you have many partitions or very high throughput. On the consumer side, fetch.min.bytes and fetch.max.wait.ms control how eagerly the consumer fetches — increasing fetch.min.bytes tells the broker to wait until it has at least N bytes of messages before responding, reducing the number of fetch requests.

Connect the settings to their mechanism: “linger.ms trades latency for batch size; compression trades CPU for bandwidth — understand the constraint you are optimizing for.”

Gotcha follow-up: Why can increasing max.in.flight.requests.per.connection improve throughput, and what is the risk? max.in.flight.requests.per.connection (default 5 with idempotent producers) controls how many unacknowledged requests the producer can have outstanding to a single broker simultaneously. Increasing this allows more pipelined requests — the producer does not wait for request N to be acknowledged before sending request N+1. This improves throughput on high-latency networks because network round-trips are overlapped. The risk: without idempotent producers, if request 2 is retried after request 3 has already been written, messages can be written out of order — request 3 appears before request 2 in the partition log. With enable.idempotence=true, the broker enforces per-partition sequence numbers, so out-of-order writes are rejected with an OutOfOrderSequenceException and the producer fails. With idempotent producers, max.in.flight.requests.per.connection is safely capped at 5 — beyond 5, the sequence number tracking overhead limits additional benefit anyway.

Q9: What is the difference between KStream and KTable in Kafka Streams? Concept Check

One-line answer: KStream is an unbounded append-only event log where every record matters; KTable is a compacted changelog representing current state — only the latest value per key is relevant.

Full answer: KStream and KTable are the two fundamental abstractions in the Kafka Streams DSL, representing two different interpretations of a Kafka topic. A KStream represents a stream of events where every individual record is significant — it is append-only and unbounded. Think of it like a database transaction log: every INSERT, UPDATE, and DELETE is a separate event. You process each event independently. A null value in a KStream means a message with a null payload — potentially valid. A KTable represents the current state of a dataset, keyed by a primary key — like a database table’s current snapshot. Only the latest value for each key matters: when a new record arrives for key “user-123”, it replaces the previous value in the table. A KTable is backed by a local state store (RocksDB) that Kafka Streams maintains per application instance. A null value in a KTable is a tombstone — it deletes the key from the table entirely, used for cleanup and compaction. The conversion between them: KStream.toTable() converts a stream to a table by retaining only the latest value per key — effectively collapsing the event history. KTable.toStream() converts a table back to a stream, emitting a record for every update to the table. They join differently: KStream-KTable join enriches each stream event with the current table state at the time of the event (low-latency lookup); KStream-KStream join matches events within a time window; KTable-KTable join produces a new table representing the join of the two tables’ current states.

Use a concrete example: “A stream of order events is a KStream — every order matters. The current inventory count per product is a KTable — only the latest count matters, not every increment.”

Gotcha follow-up: How does Kafka Streams handle the case when a KTable is too large to fit in local memory? Kafka Streams uses RocksDB as the local state store backend — an embedded key-value store that stores data on disk, not just in memory. RocksDB has a configurable block cache (in-memory) and stores the rest on disk, so a KTable that is too large for memory spills to disk automatically. The state stores are backed by changelog topics in Kafka: every write to a state store is also written to the changelog topic. On application restart, the state store is restored by replaying the changelog topic — this can take time for large stores. To minimize restore time: use standby replicas (num.standby.replicas configuration) — Kafka Streams keeps shadow copies of state stores on other application instances, so failover is fast (seconds instead of minutes for large stores). Interactive Queries allow other services to query a Kafka Streams application’s state store directly via REST, enabling the state to be used as a queryable data layer.

Q10: What is the Schema Registry, and how does Avro enable schema evolution? Concept Check

One-line answer: The Schema Registry stores Avro schemas centrally, assigns each a numeric ID that is prefixed to every message, and enforces compatibility rules so producers and consumers can evolve schemas independently without breaking each other.

Full answer: Without a Schema Registry, producers and consumers must agree on message format through out-of-band documentation and human coordination — fragile and error-prone. The Schema Registry (Confluent or AWS Glue) provides centralized schema management with enforced compatibility. Here is how it works mechanically: when a producer wants to send its first message with a given schema, it registers the Avro schema with the Registry and receives back a 4-byte integer schema ID. Every subsequent message sent by the producer has a 5-byte prefix: one magic byte (0x00) followed by the 4-byte schema ID, followed by the Avro-serialized payload. When a consumer receives a message, it reads the 4-byte schema ID from the prefix, fetches the corresponding schema from the Registry (cached locally after the first fetch), and uses it to deserialize the payload. This enables schema evolution. Backward compatibility — the default and recommended mode — means a new schema can read data written by an old schema. This allows: adding optional fields with default values (new consumers read new and old data; old consumers reading new data see the default value for the new field), and removing optional fields. It forbids: adding required fields without defaults (old producers do not write them), renaming fields, and changing field types. Forward compatibility means old consumers can read data written by new producers. Full compatibility requires both. Always design schemas for backward compatibility so you can roll out producer changes before consumer changes — the typical deployment order.

Emphasize the byte-level message format — magic byte + schema ID prefix — as it shows you understand the mechanics, not just the concept.

Gotcha follow-up: What happens if a producer publishes a schema that is incompatible with the current version? The Schema Registry rejects the schema registration attempt with an HTTP 409 Conflict response containing a message explaining the compatibility violation — for example, “field ‘email’ removed without a default value, violating BACKWARD compatibility.” The producer’s Avro serializer receives this rejection and throws an exception, preventing the incompatible message from ever reaching the topic. This is the contract enforcement point: the Registry acts as a gatekeeper that prevents schema violations from reaching consumers. The producer developer must fix the schema change to be compatible — for example, keep the removed field with a default value for the transition period, or plan a coordinated consumer-first migration for a breaking change. This fail-fast behavior at registration time is far better than consumers silently failing to deserialize messages in production.

Q11: When would you choose Kafka over RabbitMQ over SQS? Tradeoff Question

One-line answer: Kafka for high-throughput streaming with replay capability; RabbitMQ for complex routing and traditional task queues; SQS for AWS-native serverless workloads where operational simplicity outweighs streaming features.

Full answer: These three are message systems that solve similar but distinct problems. Kafka is a distributed event log, not a traditional message broker. Its defining characteristic is retention: messages are stored on disk for a configurable period — days or weeks — and consumers can replay from any offset. This enables multiple consumer groups to independently consume the same topic at their own pace, event sourcing patterns, audit logging, and CDC pipelines. Throughput is in the millions of messages per second across a cluster. The trade-off: Kafka requires significant operational expertise to run well (partition sizing, replication, ISR monitoring) or a managed service (Confluent Cloud, AWS MSK). Choose Kafka when you need replay capability, high throughput (>10,000 messages/second), multiple independent consumers of the same event stream, or event sourcing. RabbitMQ is a traditional AMQP message broker with sophisticated routing: exchanges route messages to queues using binding keys, fanout exchanges broadcast to all queues, topic exchanges route by pattern. Messages are typically delivered to one consumer and deleted after processing. Throughput is in the tens of thousands per second — less than Kafka but with richer routing semantics. Choose RabbitMQ for task distribution (work queues), complex routing requirements, or when you need AMQP protocol compliance. AWS SQS is a fully managed, serverless queue. No infrastructure to operate, no servers to provision, native integration with Lambda, SNS, and other AWS services. FIFO queues provide ordering and exactly-once delivery within a message group. No replay — consumed messages are deleted. Choose SQS for AWS-native applications where operational simplicity is the priority and Kafka’s streaming capabilities are not needed.

Use a decision matrix framing: “Kafka for stream + replay; RabbitMQ for routing + AMQP; SQS for serverless + simplicity.”

Gotcha follow-up: Can Kafka replace a traditional job queue (work queue) pattern? Yes, but with caveats. Kafka can distribute work across consumers in a consumer group — each partition is consumed by one consumer, naturally distributing load. For N consumers, use at least N partitions. The difference from RabbitMQ’s work queue: in Kafka, messages are not deleted after consumption — they are retained. This means if a consumer group processes messages, another consumer group can independently replay the same messages for a different purpose (audit, analytics). In RabbitMQ, once a message is acknowledged it is gone. The Kafka limitation for work queues: if some work items take longer than others, a consumer stuck on a slow item will not pick up new items for its assigned partitions while other partitions’ consumers are idle — Kafka lacks the dynamic per-message routing that RabbitMQ uses to balance load across workers. Kafka works well as a work queue when work items are roughly uniform in processing time; for highly variable work items, RabbitMQ’s dynamic dispatch is a better fit.

Q12: What are ISR and min.insync.replicas, and how do they protect against data loss? Concept Check

One-line answer: ISR (In-Sync Replicas) is the set of replicas caught up with the leader; min.insync.replicas defines how many must acknowledge writes — together they ensure data survives broker failures without loss.

Full answer: ISR stands for In-Sync Replicas — the set of partition replicas (including the leader) that are currently caught up with the leader’s log within a configured lag threshold: replica.lag.time.max.ms, which defaults to 30 seconds. A follower that falls behind this threshold is removed from the ISR — it is considered “out of sync.” When the follower catches up again, it is re-admitted to the ISR. The ISR changes dynamically based on replica performance and network conditions. min.insync.replicas is a broker or topic-level setting that defines the minimum number of replicas (including the leader) that must acknowledge a write when acks=all. With replication.factor=3 and min.insync.replicas=2: a write is committed only when at least 2 replicas have written it. If the leader fails, the controller elects a new leader from the ISR — and since any ISR member has the latest committed data, no data is lost. This configuration tolerates one broker failure with zero data loss. The protection mechanism: if the ISR shrinks below min.insync.replicas — for example, two brokers crash leaving only one in the ISR — the leader rejects new writes with NotEnoughReplicasException rather than acknowledging writes that only one replica holds. The partition is protected from data loss at the cost of write availability. Monitor this with the under-replicated-partitions metric in JMX or Prometheus — alert when any partition’s ISR count falls below its replication factor.

Always explain the protection mechanism both ways: what happens when it’s healthy (writes succeed) and what happens when ISR shrinks below min.insync.replicas (writes are rejected).

Gotcha follow-up: Can you ever have a situation where all replicas are in the ISR but still lose data? Yes — if you use acks=1 instead of acks=all, the leader writes the message and acknowledges it to the producer before any follower replicates it. If the leader crashes at that moment — after acknowledgment but before the followers pull the message — the message is lost. The ISR is irrelevant here because acks=1 does not wait for ISR acknowledgment. Also: if the entire cluster loses power simultaneously, messages that were acknowledged by acks=all but not yet flushed to disk (still in the OS page cache) can be lost — this is the log.flush.interval.messages and log.flush.interval.ms configuration. In practice, Kafka relies on replication for durability rather than fsyncing every message to disk (which would be prohibitively slow), so a simultaneous multi-broker crash is the one scenario where data loss is possible even with acks=all. This is documented and accepted in Kafka’s durability model.

Q13: What is the difference between retention-based and log compaction in Kafka? Concept Check

One-line answer: Retention deletes old messages by age or size — keeping a time window of events; log compaction keeps only the latest value per key indefinitely — providing a compact current-state snapshot of every key ever written.

Full answer: Kafka topics have two distinct cleanup policies that serve completely different use cases. Retention-based cleanup — cleanup.policy=delete, the default — deletes log segments when either retention.ms (default 7 days) is exceeded or the total log size exceeds retention.bytes. Kafka deletes the oldest log segments, so the topic always contains a rolling window of recent messages. This is ideal for event streams where old data has no value after a time window: application logs, user activity events, sensor readings. Log Compaction — cleanup.policy=compact — keeps only the most recent message for each key. The compaction process, run in the background by the log cleaner threads, identifies duplicate keys in older log segments and removes all but the latest value. A message with a null value is a tombstone — it signals “delete this key from the compacted log.” After the tombstone has been propagated and sufficient time has passed (delete.retention.ms), the tombstone itself is removed, cleaning up the key entirely. The result: the compacted topic contains exactly one message per key — the latest state. This topic can be fully replayed to reconstruct the current state of every key, but it is much smaller than a full event log. Use cases: Kafka Streams’ KTable changelog topics (reconstruct local state stores on restart), CDC topics (current state of every database row), entity state topics (current user profile, current product inventory). You can combine both: cleanup.policy=compact,delete — keeps the latest value per key but also discards keys older than retention.ms.

Explain the tombstone mechanism explicitly — it is the subtlety that separates candidates who truly understand compaction from those who have only read the definition.

Gotcha follow-up: If log compaction keeps only the latest value per key, is the compacted topic guaranteed to be up-to-date when a consumer reads it? No — log compaction is a background process, not a synchronous operation. The “tail” of the log (recent messages) is never compacted — compaction only runs on older log segments. This means a consumer reading a compacted topic will see all recent uncompacted messages (possibly multiple values per key from the recent tail) plus the compacted history. Consumers must apply their own deduplication or last-value semantics: a KTable in Kafka Streams automatically handles this by keeping only the latest value per key in its state store. For non-Streams consumers, you should not assume a compacted topic has exactly one message per key at any given instant — only that after compaction completes, old segments will have at most one message per key.

Q14: How do you monitor and diagnose high consumer lag in Kafka? Design Scenario

One-line answer: Consumer lag measures how far behind the consumer group is from the latest messages; monitor it with kafka-consumer-groups.sh or Prometheus, alert on threshold breaches, and fix by adding partitions, optimizing processing, or scaling consumers.

Full answer: Consumer lag is the delta between the latest offset written to a partition (the “log end offset”) and the last offset committed by the consumer group (the “committed offset”). A lag of 0 means the consumer is fully caught up. A lag of 10,000 means the consumer has 10,000 unprocessed messages queued. Growing lag indicates the consumer cannot keep up with the producer’s write rate — a potential SLA breach. To check lag manually: kafka-consumer-groups.sh –bootstrap-server localhost:9092 –describe –group my-group prints the current lag per partition, the committed offset, and the log end offset. For production monitoring: Kafka’s JMX metrics expose records-lag-max per consumer; kafka-lag-exporter (Lightbend) or the Confluent Monitoring Interceptor exports lag to Prometheus. Create a Grafana dashboard with lag time series per consumer group and topic. Set alerts: if lag exceeds a threshold (for example, 10,000 messages or 5 minutes of lag at current production rate) and is trending upward, page the on-call team. Common causes and fixes: processing is too slow per message — profile and optimize the processing code or use async processing with a separate thread pool; the partition count limits parallelism — you cannot have more consumers than partitions, so add partitions and proportionally more consumer instances; max.poll.records is set too high — each poll returns a large batch that takes too long to process, causing poll interval timeouts and rebalances; or a poison pill message blocks one partition — use DLT handling to unstick it.

Structure your answer as: how to measure, what tools to use, how to alert, and what are the common root causes with fixes.

Gotcha follow-up: Can adding more consumer instances always fix high consumer lag? No — you can only have as many active consumer instances as there are partitions. If a topic has 6 partitions and you have 6 consumer instances, adding a 7th instance gives it no partitions — it is idle. To increase parallelism beyond the current consumer count, you must first increase the partition count. However, increasing partition count has trade-offs: partitions cannot be decreased once added; more partitions means more leader elections during broker failures; Kafka’s end-to-end latency increases slightly with very high partition counts because the controller and coordinator must track more partition state. The other fix for lag without adding partitions: optimize per-message processing time. If each message takes 10ms to process and you have 6 consumers each handling one partition, your maximum throughput is 6 × 100 = 600 messages per second. Reducing processing to 1ms gives 6,000 messages per second — 10x improvement without any infrastructure changes.

Q15: How do you implement a @KafkaListener with manual commit and Dead Letter Topic handling in Spring? Design Scenario

One-line answer: Disable auto-commit with ack-mode=manual_immediate, call ack.acknowledge() only after successful processing, and configure DefaultErrorHandler with DeadLetterPublishingRecoverer to route unrecoverable failures to a DLT.

Full answer: In Spring Kafka, the default behavior uses auto-commit, which risks data loss. Here is the production configuration. In application.yml, set spring.kafka.listener.ack-mode=manual_immediate and spring.kafka.consumer.enable-auto-commit=false. In your @KafkaListener method, add an Acknowledgment parameter alongside the ConsumerRecord: the method signature becomes public void processOrder(ConsumerRecord<String, OrderEvent> record, Acknowledgment ack). Inside the method, wrap the processing logic in a try-catch. On success: call ack.acknowledge() — this commits the offset immediately. For retryable exceptions — transient failures like network timeouts, temporary DB unavailability — do not acknowledge. The message will be redelivered on consumer restart. Spring Kafka’s DefaultErrorHandler handles retry automatically with configurable backoff: it seeks back to the failed offset and retries up to maxAttempts times with exponential backoff between attempts. After maxAttempts are exhausted: for non-retryable exceptions — malformed messages, business rule violations that will always fail — the DefaultErrorHandler uses a DeadLetterPublishingRecoverer to write the failed message to the DLT (named {original-topic}.DLT by convention). The DLT message includes headers with the original topic, partition, offset, exception class, and exception message for debugging. After publishing to the DLT, the original message is acknowledged — the consumer moves past it. A separate @KafkaListener on the DLT processes failed messages for alerting, manual review, or reprocessing after fixing the root cause. Never leave the DLT unconsumed — a growing DLT is a signal of a production issue.

Mention that without DLT handling, a single bad message can permanently stall a partition — make the stakes concrete.

Gotcha follow-up: What is the difference between DefaultErrorHandler and SeekToCurrentErrorHandler in Spring Kafka? SeekToCurrentErrorHandler was the original error handler in Spring Kafka 2.x — it handled retries by seeking the consumer offset back to the failed message and reprocessing it. It was separate from the DLT publishing logic, which was handled by the DeadLetterPublishingRecoverer used with it. In Spring Kafka 2.8, DefaultErrorHandler was introduced as the unified replacement — it combines the retry-with-backoff behavior and the recoverer interface in one class, replacing SeekToCurrentErrorHandler. DefaultErrorHandler accepts a BackOff (FixedBackOff or ExponentialBackOffWithMaxRetries) and a ConsumerRecordRecoverer (typically DeadLetterPublishingRecoverer). SeekToCurrentErrorHandler is still available but deprecated as of Spring Kafka 2.8. In any new Spring Boot 3 / Spring Kafka 3.x project, use DefaultErrorHandler.

Common Mistakes:

  • Using auto-commit with critical data → auto-commit commits offsets on a timer, not after processing. Use manual_immediate ack mode for any data where loss is unacceptable.
  • Not handling the poison pill → without DLT routing, a single bad message stalls the entire partition forever. Always configure DefaultErrorHandler with DeadLetterPublishingRecoverer.
  • Setting acks=all without min.insync.replicas=2 → acks=all with a default min.insync.replicas=1 still allows data loss if the leader fails before replication. Always pair them.
  • Using eager rebalancing (default) → add CooperativeStickyAssignor to avoid “stop the world” consumer pauses. This is a real production default that should be changed.
  • Expecting ordering guarantees across partitions → Kafka only guarantees order within a single partition. Use consistent key hashing to ensure related messages go to the same partition.
  • Ignoring consumer lag monitoring → a growing lag is a silent SLA breach. Set up Prometheus + Grafana alerts on lag before going to production.

Quick Revision: acks=all + min.insync.replicas=2 = no data loss; idempotent producer = no producer-side duplicates per session; manual commit after processing = at-least-once; read_committed + transactions = exactly-once; cooperative rebalancing = no “stop the world” pauses.

Section 4: Redis & Caching

Reading guide: 15 questions covering cache patterns, failure modes, data structures, persistence, and Spring integration. Focus on the three failure modes (stampede, penetration, avalanche) and their distinct solutions — interviewers love testing that you can tell them apart.

Q1: What are the main cache write patterns — Cache-Aside, Write-Through, and Write-Behind — and when would you choose each? Concept Check

One-line answer: Cache-Aside lazily loads on miss; Write-Through keeps cache always fresh by writing synchronously; Write-Behind batches writes asynchronously for low write latency.

Full answer: I think of these three patterns as different answers to the question “when does data get into the cache and when does it leave?” With Cache-Aside, also called Lazy Loading, the application is in charge — on a read, it checks the cache first, and on a miss it fetches from the database, stores the result in the cache, then returns it. Writes simply invalidate or update the cache entry after writing to the database. The advantage is that only data that is actually requested ever enters the cache, so memory is used efficiently, and a cache failure degrades gracefully because the application can always fall back to the database. The downside is a cold-start penalty on the first request for any key, and a window of inconsistency between when the database is written and when the cache is updated. Write-Through flips this: every write to the database is also written to the cache synchronously in the same operation. The cache is always up to date, so there are no misses immediately after a write, but write latency increases because we are writing to two places, and the cache can fill up with data that is written but never read — cache pollution. Write-Behind, also called Write-Back, writes to the cache first and then flushes to the database asynchronously on a schedule or in batches. This gives very low write latency and allows batching database writes for efficiency, but introduces the risk of data loss if the cache crashes before the flush completes, and recovery logic is complex.

Lead with the analogy “the application is in charge” for Cache-Aside — it signals you understand the pattern, not just memorized the name.

Gotcha follow-up: In Cache-Aside, after you write to the DB do you update the cache or delete the cache entry? Delete (invalidate) is safer. If you update the cache synchronously after a DB write, you risk a race condition: two concurrent writes can store an old value over a newer one. By deleting the entry, the next read is a guaranteed miss that fetches the latest DB value. This is the “delete on write” pattern — the small cost of one extra DB read after every write buys you correctness.

Q2: What is a cache stampede, also called the thundering herd problem, and how do you prevent it? Concept Check

One-line answer: A popular key expires and thousands of concurrent requests simultaneously miss the cache and all query the database at once, potentially overwhelming it.

Full answer: A cache stampede happens when a popular, heavily-requested cache key reaches its TTL — its time-to-live, the expiry timer — and expires. At that moment, all of the requests that arrive before the cache is repopulated all find a miss and all independently decide to fetch from the database. If that key was receiving, say, ten thousand requests per second, the database suddenly absorbs all ten thousand queries simultaneously instead of the cache absorbing them. The four main defenses address the problem at different layers. A mutex lock means only the first thread that detects a miss acquires a distributed lock — for example, a Redis SET NX (set if not exists) — fetches from the database, repopulates the cache, and releases the lock; all other threads either wait briefly or return a slightly stale value. Probabilistic early expiration, also called XFetch, randomly triggers a cache refresh slightly before the TTL expires, spreading the refresh load over time so expiry is never a sudden cliff. A background refresh job proactively refreshes hot keys on a schedule before they expire, and serves stale data while the refresh runs — the user never sees a miss. The most robust solution for the hottest keys is logical expiration: the cache key never has a Redis TTL at all, so it never expires on its own. Instead, the expiry timestamp is stored inside the cached value. When a reader sees the logical expiry has passed, it tries to acquire a lock to refresh, but all other readers continue serving the stale value — no thundering herd, ever.

Mention all four solutions by name, then dive into whichever one the interviewer picks. Logical expiration surprises most people.

Gotcha follow-up: How does mutex lock prevent stampede if 10,000 threads are waiting on the lock? You should combine mutex with a stale-while-revalidate strategy. Threads that fail to acquire the lock immediately return the stale cached value — or a pre-computed fallback — rather than blocking. Only the lock holder fetches fresh data. This way, users see a slightly stale response for a fraction of a second, but the DB sees only one query instead of ten thousand.

Q3: What is cache penetration and how do you defend against it? Concept Check

One-line answer: Repeated requests for keys that exist in neither the cache nor the database bypass the cache entirely and hammer the database with guaranteed misses.

Full answer: Cache penetration is different from a stampede or avalanche because the problem is not about expiry — it is about data that does not exist at all. An attacker, or even just a buggy client, repeatedly queries IDs or keys that were never in the database. Every single request misses the cache (there is nothing to cache) and hits the database (which returns nothing), so the database bears the full load of every request. A common attack pattern is sending sequential non-existent user IDs in high volume. The first defense is a Bloom filter — a probabilistic data structure, meaning it can say “this key is definitely not in the database” with zero false negatives (if the Bloom filter says absent, it is truly absent), though it has a small configurable false positive rate (it occasionally says “might exist” for something that doesn’t, causing one unnecessary DB query, which is acceptable). The Bloom filter sits in front of everything: if it returns absent, reject the request immediately without touching the cache or database. The second defense is caching null values: when the database returns nothing for a key, store a null sentinel or an empty marker in the cache with a short TTL, say 30 to 60 seconds. Subsequent requests for the same non-existent key hit the cache and get the null back immediately, never reaching the database. The short TTL ensures that if the data is eventually created, the null entry expires and the next request fetches the real data.

The Bloom filter answer is a strong signal — it shows you know probabilistic data structures, not just “cache the null.”

Gotcha follow-up: What is the downside of caching null values? Two risks: first, the cache fills with null entries for non-existent keys — a memory concern. Mitigate with short TTLs and a max cache size. Second, if a key is created in the database while a null is cached, users see stale null for up to the TTL duration. Mitigate by explicitly evicting the null entry when the record is created.

Q4: What is cache avalanche and how does it differ from a stampede? Concept Check

One-line answer: A stampede is one hot key expiring; an avalanche is many keys expiring simultaneously, causing a mass miss wave that can collapse the database.

Full answer: Cache avalanche happens when a large number of cache keys expire at the same time — this is common when you pre-warm a cache at startup and set the same TTL for all keys, so they all expire in the same moment. The result is a wave of cache misses all hitting the database simultaneously, which can overload it in a way that looks like a DDoS. The primary fix is TTL jitter: instead of giving every key the same TTL, add a random offset — something like TTL equals baseTTL plus a random value between zero and some jitter range. This spreads expirations across a time window instead of concentrating them at one instant. Staggered cache warming at startup — preloading cache in waves rather than all at once — also helps because the keys are not all created simultaneously and therefore will not expire simultaneously. If an avalanche does occur and the database gets overwhelmed, a circuit breaker pattern can limit the damage: the circuit breaker monitors database error rates and, when they exceed a threshold, it trips to OPEN state and immediately returns errors or fallback responses rather than queuing thousands of database connections that compound the problem. An L2 cache — a fast, local in-process cache like Caffeine sitting in front of Redis — absorbs some reads even during a Redis miss wave, because the local cache has its own independent TTLs.

The key to impressing here: clearly distinguish stampede (one key), avalanche (many keys), and penetration (non-existent keys). Interviewers use these interchangeably to see if you notice.

Gotcha follow-up: If you add jitter, how much jitter is appropriate? It depends on your read traffic pattern. A common rule of thumb is jitter of 10–20% of the base TTL. If your base TTL is 10 minutes, add random(0, 60–120 seconds). This spreads your expirations across 2 minutes instead of one instant, which is usually enough to prevent a mass hit. Too much jitter means some keys expire too early and get evicted while still useful.

Q5: What is cache breakdown and how is it different from a stampede? Concept Check

One-line answer: Cache breakdown is a stampede on a single extremely hot key — the distinction is that breakdown focuses on one critical key, and logical expiration is the canonical fix.

Full answer: Cache breakdown is conceptually a stampede, but the distinction interviewers draw is this: a stampede is typically described as many different popular keys expiring at similar times, while breakdown specifically refers to one single, extremely hot key — imagine the homepage data for a major website, or a leaderboard entry — expiring while thousands of concurrent requests are all asking for that exact key. The simultaneous cache misses for that one key all race to the database. The mutex lock solution works: only the first thread to detect a miss acquires a Redis lock, fetches from the database, repopulates the cache, and releases the lock; all others wait. But the more elegant solution for the single hottest key is logical expiration: you deliberately never set a Redis TTL on the key, so it never expires and a thundering herd is structurally impossible. Instead, you embed the expiry timestamp inside the cached value itself. When a reader fetches the value and sees that the embedded timestamp is in the past, it tries to acquire a background-refresh lock. If it wins the lock, it refreshes the value and releases the lock. If it does not win the lock, it simply returns the slightly stale value to the user. The result is that users always get a value — either fresh or stale by a fraction of a second — and the database sees at most one refresh query at a time.

The phrase “structurally impossible” for logical expiration shows you understand WHY it works, not just what it does.

Gotcha follow-up: With logical expiration, how do you handle a cold start where the key has never been set? On cold start, the key is absent entirely — a true miss, not a stale hit. You need an initialization path: typically, at application startup or on first miss, one thread acquires the lock and does a synchronous fetch, populating the cache with logical expiry set to now plus TTL. All other threads wait for that first population. This is the only time users wait; after that, logical expiration ensures they always get a value.

Q6: What are the Redis eviction policies and when would you choose each one? Concept Check

One-line answer: Eviction policies tell Redis what to delete when memory is full; the right policy depends on whether Redis holds a pure cache, persistent data, or a mix.

Full answer: When Redis is configured with a maxmemory limit and a new write would exceed it, the eviction policy determines which existing key gets deleted to make room. Understanding each policy requires knowing whether your keys are a pure cache — all data is safely in a database too — or permanent data that cannot be lost. noeviction returns a write error and evicts nothing; use this when Redis holds data that must never be discarded, though your application must handle the error. allkeys-lru evicts the least recently used key from all keys regardless of TTL — this is the best general-purpose policy for a pure cache, because it prioritizes keeping recently-accessed data, which correlates with what users are likely to ask for again. volatile-lru applies LRU eviction only to keys that have a TTL set, leaving keys without TTL untouched — use this when Redis holds a mix of ephemeral cache entries (which have TTLs) and permanent data (which do not), so you never accidentally evict your permanent data. allkeys-lfu evicts the least frequently used key from all keys — LFU, which stands for Least Frequently Used, counts access frequency rather than recency, making it better than LRU when your access pattern is heavily skewed, meaning a small number of very hot keys get the vast majority of traffic and should never be evicted even if not accessed in the last few seconds. volatile-ttl evicts the key with the shortest remaining TTL first — useful when you want Redis to naturally prioritize expiring soon-to-expire keys early rather than wasting memory on them. allkeys-random evicts a random key — suitable only when all keys are equally likely to be needed and you have no recency or frequency preference.

Always say “pure cache vs. mixed store” to frame the decision — it’s the key insight interviewers want.

Gotcha follow-up: When would allkeys-lfu outperform allkeys-lru? Imagine a leaderboard or product catalog where a handful of top-10 items receive millions of requests, but each burst of requests has gaps — the same top-10 items are accessed in waves. LRU might evict them during quiet gaps. LFU tracks long-term access frequency and keeps the high-frequency items regardless of recency. Skewed, long-tail traffic distributions with clear hot spots favor LFU.

Q7: What are the main Redis data structures and what problems does each solve best? Concept Check

One-line answer: Redis has seven core data structures beyond simple key-value strings, each optimized for a specific access pattern — picking the right one eliminates entire layers of application logic.

Full answer: I think of Redis data structures as pre-built solutions for common problems. String is the simplest — it stores any binary-safe value up to 512 MB, and the INCR command makes it an atomic counter (atomic means it increments in a single CPU operation with no race condition), making it perfect for page view counters, rate limiting counters, and distributed locks using SET key value NX PX (set if not exists, with expiry in milliseconds). Hash stores a map of field-to-value pairs for a single key — HGET user:123 email fetches just one field without deserializing an entire object, making it ideal for storing user profiles or session data where you only need individual fields. List is a doubly-linked list supporting LPUSH (push to left/head) and RPOP (pop from right/tail), which gives you a queue; it is also useful for LRANGE to fetch recent items like the last twenty notifications. Set stores unique, unordered members — it supports set operations like SINTER (intersection) for “common followers between user A and user B” and SADD for tagging systems. Sorted Set is one of Redis’s most powerful structures: every member has a numeric score, and members are always ordered by score; ZADD leaderboard 1500 userId123, ZRANGE leaderboard 0 9 REV for the top ten — also useful for rate limiting (score = timestamp) and delayed task queues (score = scheduled execution time). HyperLogLog is a probabilistic data structure — it estimates the count of unique elements with only 0.81% error using a fixed 12 KB of memory regardless of cardinality; PFADD and PFCOUNT are perfect for counting unique daily active users without storing every user ID. Stream is a persistent, ordered event log inside Redis supporting consumer groups — groups of consumers that split work among themselves with acknowledgment semantics — making it like a lightweight Kafka for pub-sub with delivery guarantees.

Mention Sorted Set for delayed queues and HyperLogLog — both signal depth and usually surprise the interviewer.

Gotcha follow-up: When would you prefer Redis Streams over Pub/Sub? Pub/Sub is fire-and-forget — if a subscriber is offline when a message is published, it is gone forever. Streams persist every message until trimmed and support consumer groups with ACK-based at-least-once delivery. Use Pub/Sub for real-time dashboards where missing a message is harmless; use Streams when you need reliability, replay, or fan-out to competing consumers.

Q8: How does Redisson’s RLock implement a distributed lock, and what is the watchdog mechanism? Concept Check

One-line answer: Redisson uses an atomic Lua script for lock acquisition and a background watchdog thread that automatically extends the TTL to prevent expiry during slow operations.

Full answer: A distributed lock ensures that across an entire cluster of JVM instances, only one thread at a time can execute a critical section — for example, deducting inventory or sending a scheduled email. Redisson’s RLock implements this on top of Redis. When a thread calls lock.lock(), Redisson runs a Lua script atomically on the Redis server — atomic here means the entire script executes as one indivisible operation, with no other command able to interleave. The script does a SET NX (set if not exists) with a TTL, and also stores the thread ID so the same thread can re-enter the lock (reentrant lock behavior). The watchdog is the critical reliability mechanism: it is a background thread started automatically when you call lock() without an explicit lease time. Every 10 seconds (by default, one-third of the default 30-second lock TTL), the watchdog sends a keepalive command to Redis extending the TTL by another 30 seconds, as long as the owning JVM process is still alive. This prevents the lock from expiring in the middle of a long database transaction or a slow external API call. If the JVM crashes, the watchdog stops, the TTL is not renewed, and the lock expires automatically — preventing a deadlock. You should always release the lock in a finally block so it is released even if an exception occurs: try { lock.lock(); doWork(); } finally { lock.unlock(); }. Use tryLock(waitTime, leaseTime) when you want an explicit expiry and to disable the watchdog.

The watchdog story is what separates someone who used Redisson from someone who truly understands it.

Gotcha follow-up: What happens if the Redis master crashes after you acquire the lock but before the replica syncs? This is the fundamental weakness of single-instance Redis locks. The lock data is lost, another instance can acquire the lock from the new master, and you have two holders simultaneously. This is the problem Redlock was designed to solve — see Q9. For most business applications, the brief window of dual lock holding is acceptable. For safety-critical operations, use ZooKeeper or etcd which offer strong consistency guarantees.

Q9: What is the Redlock algorithm and why is it controversial? Tradeoff Question

One-line answer: Redlock acquires a distributed lock from a majority of independent Redis instances for fault tolerance, but critics argue clock drift and process pauses make it unsafe for safety-critical use.

Full answer: Redlock, proposed by Redis creator Salvatore Sanfilippo (antirez), tries to solve the single-master lock weakness: if you have N independent Redis instances with no replication between them, you acquire the lock by trying to SET the same key on all N instances simultaneously. The lock is valid only if you succeed on a strict majority — N/2 plus one — AND the total time taken to acquire from the majority is less than the lock’s TTL, meaning the lock is still valid on all those instances. The release logic deletes the key from all instances. The controversy was sparked by Martin Kleppmann, a distributed systems expert, who argued that Redlock makes an implicit assumption that clocks are reasonably synchronized across machines — NTP, which is the protocol that synchronizes server clocks, can cause jumps or drift. More critically, a process pause — such as a garbage collection stop-the-world event in a JVM — can cause a thread to believe it holds a valid lock while the lock has actually expired on all Redis instances, allowing another client to acquire it. This is the dual-write safety violation. Antirez disagreed, arguing the algorithm is safe within its documented assumptions. My practical take: use Redisson RLock for most production distributed locking, because it is battle-tested and the risk window is very small for typical business operations. For financial, billing, or safety-critical operations where two processes simultaneously acting is genuinely catastrophic, use ZooKeeper or etcd, which use the Raft or Zab consensus algorithms and provide strong consistency guarantees without clock assumptions.

Showing you know both sides of the Martin vs antirez debate signals genuine distributed systems knowledge.

Gotcha follow-up: What is fencing token and why is it considered a better solution than Redlock? A fencing token is a monotonically increasing number returned with every successful lock acquisition — like a version number. When you write to a resource, you include your fencing token, and the resource server rejects any write with a token older than the last seen. Even if two clients both believe they hold the lock simultaneously, only the one with the higher token succeeds, making the resource write idempotent. This works regardless of clock drift and process pauses. Kleppmann argues fencing tokens, implemented using a system with monotonic counters like ZooKeeper’s zxid, are safer than Redlock.

Q10: How does Redis Cluster work — hash slots, routing, and MOVED vs ASK redirects? Design Scenario

One-line answer: Redis Cluster shards the key space into 16,384 hash slots distributed across master nodes; clients route directly to the correct node using a cached slot map, and MOVED vs ASK tells the client whether a slot has permanently migrated or is temporarily in transit.

Full answer: Redis Cluster solves horizontal scaling by sharding data across multiple master nodes instead of a single server. The entire key space is divided into exactly 16,384 hash slots — you can think of these as 16,384 buckets. For any given key, the slot is computed as CRC16(key) mod 16384, where CRC16 is a checksum algorithm that produces a deterministic number from any string. Each master node owns a contiguous range of slots — for example, in a 3-master cluster, node A might own slots 0 to 5460, node B 5461 to 10922, and node C 10923 to 16383. Smart cluster clients like Jedis and Lettuce request the cluster’s slot-to-node mapping at startup and cache it locally. When you send a command, the client computes the slot, looks up the correct node in its cache, and sends the command directly to that node without an intermediate proxy hop. If the client’s cached mapping is stale — for example, a slot was migrated to a different node — the target node responds with MOVED slot ip:port, which is a permanent redirect meaning the slot now permanently lives at that address; the client updates its cache. If a slot is in the middle of being migrated and the key specifically has not moved yet, the node responds with ASK ip:port, which is a temporary redirect for this one request only — the client should send ASKING to the target node first, then retry the command, but not update its slot cache. The minimum production deployment is three masters each with one replica, giving six nodes total — this survives one master failure per shard. Operations on keys in different slots — like MGET with keys that hash to different nodes — fail in cluster mode. The solution is hash tags: wrapping part of the key in curly braces, like {user:123}:orders and {user:123}:preferences, forces Redis to use only the text inside the braces for the slot calculation, ensuring both keys land on the same node.

MOVED vs ASK is a classic deep-dive question — permanent vs temporary is the one-sentence distinction.

Gotcha follow-up: How does Redis Cluster handle a master failure? Each master has one or more replica nodes that replicate data asynchronously. When a master becomes unreachable, the cluster runs an internal election using a quorum vote among the remaining masters. If a majority of masters agree the node is down, one of its replicas is promoted to master. This typically takes 10–30 seconds. During that window, the affected slots are unavailable — Redis Cluster does not guarantee availability during failover, it guarantees that once a new master is elected, the cluster resumes correctly.

Q11: What is the difference between RDB and AOF persistence in Redis, and what should you use in production? Tradeoff Question

One-line answer: RDB takes periodic binary snapshots for fast recovery but risks losing minutes of data; AOF logs every write command for near-zero data loss but recovers more slowly; production should use both.

Full answer: RDB stands for Redis Database — it is a point-in-time snapshot of the entire dataset written to a binary file. The BGSAVE command forks a child process — forking means creating a copy of the current process — that writes the snapshot to disk while the main Redis process continues serving requests. Forking uses copy-on-write at the OS level, so memory is not actually duplicated unless the main process writes to it. RDB is fast to restore because the binary file is compact and Redis can load it in seconds. The risk is data loss: if Redis crashes between snapshots, you lose all changes since the last snapshot, which could be minutes. AOF stands for Append-Only File — every write command that Redis executes is appended to a log file in text format. On restart, Redis replays every command in order to reconstruct the dataset. The appendfsync setting controls durability: always flushes after every command for zero data loss but lowest throughput; everysec (the default) flushes once per second, meaning at most one second of data can be lost in a crash, which is a very good tradeoff for most applications; no lets the OS decide when to flush, which is fastest but least safe. The AOF file grows indefinitely and must be periodically rewritten — Redis compacts it by replacing the log with the minimum set of commands needed to reproduce the current state. Recovery from AOF is slower than RDB because Redis replays commands rather than loading a binary snapshot. The production recommendation is to enable both: AOF provides durability and near-zero data loss; RDB provides fast disaster recovery snapshots. If Redis crashes during a run, it will use the AOF file to recover (more complete), but for a fresh restore from backup, RDB is faster.

“Use both” with the reasoning is the right answer — saying only AOF or only RDB signals incomplete production experience.

Gotcha follow-up: What is AOF rewrite and when does it trigger? Over time, the AOF file accumulates redundant commands — for example, a counter that was incremented a million times has a million INCR entries, but only the final SET to the current value is needed. AOF rewrite rewrites the file from scratch using the current in-memory state, replacing all those commands with minimal equivalents. It is triggered automatically when the AOF file grows beyond a threshold, configurable via auto-aof-rewrite-min-size and auto-aof-rewrite-percentage. It runs in a background child process and does not block the main Redis thread.

Q12: When would you use Redis Pub/Sub versus Redis Streams? Tradeoff Question

One-line answer: Pub/Sub is fire-and-forget with no persistence — great for live notifications; Streams are a persistent, replayable log with consumer groups — use when you need delivery guarantees or offline consumers.

Full answer: Redis Pub/Sub is a simple broadcast mechanism: a publisher sends a message to a named channel, and all currently-subscribed subscribers receive it instantly. The word “currently” is the key limitation — Pub/Sub has no persistence at all. If a subscriber is offline or slow and misses a message, that message is gone forever. There is no concept of consumer groups, no acknowledgment mechanism, and no way to replay messages. This makes it perfect for real-time, ephemeral use cases like live dashboard updates where missing one data point is harmless, presence notifications (a user just came online), or chat messages where you have a separate persistent store for history. Redis Streams, introduced in Redis 5, are a fundamentally different data structure: a persistent, ordered, append-only log, conceptually similar to Apache Kafka. Each message has a unique auto-generated ID and is stored until explicitly trimmed with XTRIM or XDEL. XADD appends messages; XREAD reads them. Consumer groups — defined with XGROUP CREATE — allow multiple consumers to coordinate: the group distributes messages among consumers so each message is processed by exactly one consumer in the group, and each consumer must explicitly acknowledge processing with XACK. This enables at-least-once delivery semantics, since unacknowledged messages can be re-delivered. Consumers can reconnect and pick up from their last acknowledged position. Use Streams when you need reliability, fan-out to competing workers, replay capability, or integration between services where message loss is unacceptable.

The ACK mechanism is what most people forget about Streams — mention it to show you understand the delivery guarantee.

Gotcha follow-up: Can Redis Streams replace Kafka? For lower-volume use cases — say, under a few hundred thousand messages per second — Redis Streams is a viable alternative to Kafka with simpler operational overhead. However, Kafka provides stronger durability guarantees through replication across multiple brokers, longer message retention, partitioning for true horizontal parallelism, and a more mature ecosystem of connectors. Redis Streams runs in a single Redis node or cluster and does not have Kafka’s multi-broker replication model. Use Redis Streams for lightweight internal event queues; use Kafka for high-volume, multi-team, durability-critical event streaming.

Q13: How do Spring Cache annotations work — @Cacheable, @CacheEvict, and @CachePut? Concept Check

One-line answer: @Cacheable reads from cache before method execution; @CacheEvict deletes cache entries after writes; @CachePut always executes the method and updates the cache with the result.

Full answer: Spring Cache provides a declarative caching layer through annotations backed by AOP — Aspect-Oriented Programming — which means Spring wraps your bean method in a proxy that intercepts calls and handles cache logic transparently. @Cacheable is placed on read methods: before the method runs, Spring checks the cache using the key (derived from method parameters by default, or customized with a SpEL expression — Spring Expression Language). On a cache hit, Spring returns the cached value and never executes the method body. On a miss, Spring runs the method, stores the result in the cache, and returns it. The unless attribute lets you conditionally skip caching — for example, unless=“#result == null” prevents storing null results which would hide future data creation. @CacheEvict is placed on update or delete methods: after the method executes, Spring removes the specified cache entry. Use allEntries=true to evict all entries in a cache region when a bulk update makes individual invalidation impractical. @CachePut is for save or update methods where you want to both execute the method AND update the cache — it always runs the method, unlike @Cacheable which skips it on a hit, and stores the return value back into the cache. This keeps the cache fresh after every write. @Caching combines multiple cache annotations on a single method for complex scenarios. To enable: add @EnableCaching to a @Configuration class and define a CacheManager bean — for Redis use RedisCacheManager backed by a RedisConnectionFactory. Always use GenericJackson2JsonRedisSerializer for serialization rather than Java’s default serialization, which is brittle across class changes and cannot be read by other languages.

Explaining WHY @CachePut exists — always executes AND updates, vs @Cacheable which skips on hit — shows you genuinely understand the difference.

Gotcha follow-up: What happens if you call a @Cacheable method from within the same class? The same proxy bypass problem as @Transactional: because Spring’s caching uses AOP proxies, a self-call like this.getCachedData() bypasses the proxy entirely and goes directly to the method, with no cache check. The fix is to inject the bean into itself using @Autowired or @Self (Spring 6) and call via the injected reference, or extract the cached method into a separate Spring-managed bean.

Q14: How does Spring Session with Redis enable stateless horizontal scaling? Design Scenario

One-line answer: Spring Session stores HTTP session data in Redis instead of in-process memory, so any server node can serve any user’s request without sticky sessions.

Full answer: Traditional servlet-based applications store session data — like the currently logged-in user, a shopping cart, or form state — in the server’s in-process memory. This creates a problem for horizontal scaling: if a load balancer routes a user’s second request to a different server than their first, the second server has no knowledge of their session, and the user appears logged out. The typical workaround is sticky sessions — configuring the load balancer to always route a given user to the same server using a session affinity cookie. This is brittle: if that server restarts or is removed, the user loses their session. Spring Session solves this by moving session storage out of the JVM heap into an external store — Redis. Annotate a @Configuration class with @EnableRedisHttpSession and provide a RedisConnectionFactory bean. At runtime, Spring Session replaces the standard servlet HttpSession implementation: the session ID is stored in a browser cookie, and all session data is stored in Redis as a hash at the key spring:session:sessions:{sessionId}. Every application node reads from and writes to the same Redis store. Any node can handle any user’s request because the session data is in Redis, not in that node’s memory. Rolling deploys — replacing nodes one at a time — no longer lose user sessions because sessions survive the restart of any individual node. The session TTL is controlled by server.servlet.session.timeout and Redis will auto-expire the hash entry accordingly. For production, use a Redis Sentinel or Redis Cluster setup for high availability of the session store itself.

The phrase “sticky sessions are no longer needed” is the key business value — say it explicitly.

Gotcha follow-up: How do you handle session data that is too large or too complex to store in Redis? Two strategies: first, minimize what you store in the session — only store the user ID and essential claims, not large objects or entire domain models. Reconstruct full objects from the database on each request using the stored user ID. Second, use JWT-based stateless authentication instead of server-side sessions entirely — the token itself carries all necessary claims, and there is no server-side session state to store. This is the direction most modern REST APIs take.

Q15: How do you implement a sliding window rate limiter using Redis? Design Scenario

One-line answer: Store each request’s timestamp in a per-client Redis Sorted Set, remove timestamps older than the window, count the remaining entries, and allow or reject — all atomically in a Lua script.

Full answer: A rate limiter controls how many requests a client can make within a time window — for example, 100 requests per minute per user. The sliding window approach, as opposed to a fixed window which resets at the start of each minute, considers a rolling window of the last N milliseconds, so a burst right before the window boundary does not get double the limit. The implementation uses a Redis Sorted Set — a data structure where every member has a numeric score — with one key per client, like rate:user:123. The score for each entry is the request timestamp in milliseconds, and the member value can also be the timestamp (making each entry unique). For each incoming request, the algorithm is: first, remove all entries with a score older than now minus windowSizeMs using ZREMRANGEBYSCORE — this trims the sorted set to only the current window. Second, count the remaining entries with ZCARD. If the count is below the limit, allow the request: add the current timestamp with ZADD, set an EXPIRE on the key so it auto-cleans after inactivity, and proceed. If the count is at or above the limit, reject the request with a 429 Too Many Requests response. Critically, these steps — the remove, count, conditional add, and expire — must be atomic to prevent race conditions between concurrent requests. Wrap them in a Lua script that Redis executes atomically. In Spring, use RedisScript to register the Lua script and execute it via RedisTemplate. Return the current count and the time until the oldest entry expires so the client can set a Retry-After header.

Mentioning the Lua script atomicity and the Retry-After header signals production readiness.

Gotcha follow-up: What are the tradeoffs between sliding window and token bucket rate limiting? Sliding window is accurate but memory scales with request rate — each request adds an entry to the sorted set. Token bucket stores only two numbers (current token count and last refill time) per client, making it O(1) memory regardless of traffic. Token bucket also naturally allows burst up to the bucket capacity, which is user-friendly — a user who waits can accumulate tokens and then make a burst. Sliding window prevents any burst above the limit within the window. For strict rate limiting with predictable behavior, use sliding window. For APIs that want to reward patient users with burst capacity, use token bucket. Redis also supports a fixed window with a simple INCR counter, which is simplest but can allow 2x the limit at window boundaries.

Common Mistakes — Section 4:

  • Using Redis KEYS * in production → KEYS is O(N) and blocks Redis’s single main thread for the entire scan, freezing all other operations. Use SCAN with a cursor instead — it returns a small batch per call and does not block.
  • Not wrapping rate limiter logic in a Lua script → Without atomicity, two concurrent requests can both read “count = 99” and both be allowed, exceeding the limit. Lua scripts execute atomically on the Redis server.
  • Enabling Java serialization with Spring Cache → Java serialization produces brittle, version-sensitive blobs that break when you add or rename fields. Use GenericJackson2JsonRedisSerializer for human-readable, forward-compatible JSON.
  • Same TTL on all cache keys at startup → All keys expire simultaneously, causing an avalanche. Always add TTL jitter.
  • Using Pub/Sub when consumers can be offline → Pub/Sub drops messages for offline subscribers silently. Use Streams for any consumer that can restart or lag.

Quick Revision: Three failure modes — stampede (one hot key), avalanche (many keys at once), penetration (non-existent keys) — each has its own fix: mutex/logical expiry, TTL jitter, Bloom filter/cache-null.

Section 5: Security

Reading guide: 15 questions covering JWT internals, OAuth2 flows, Spring Security, and OWASP. Pay special attention to the JWT validation checklist (Q4) and the BOLA pattern (Q15) — these are the most common real-world vulnerabilities interviewers test.

Q1: What is the structure of a JWT — JSON Web Token — and what does it actually guarantee? Concept Check

One-line answer: A JWT is three Base64url-encoded sections — header, payload, signature — separated by dots; it guarantees the payload has not been tampered with, but it does NOT guarantee confidentiality.

Full answer: A JWT is a compact, self-contained token for transmitting claims between parties. It has exactly three sections separated by dots: Header.Payload.Signature. The header is a JSON object encoded in Base64url — Base64url is a variant of Base64 that is safe to use in URLs — containing the algorithm used to sign the token, like RS256 or HS256, and the token type, which is JWT. The payload is also a JSON object encoded in Base64url containing the claims: sub is the subject — typically the user ID, iss is the issuer — who created the token, aud is the audience — who the token is intended for, exp is the expiry timestamp as a Unix epoch number, iat is the issued-at timestamp, jti is a unique JWT ID that can be used to blacklist individual tokens, and any custom claims your application needs like roles or permissions. The signature is a cryptographic hash of the encoded header plus the encoded payload, computed using the algorithm specified in the header and a secret or private key. Anyone who holds the correct verification key can verify that the signature matches the content — if any character in the header or payload was changed, the signature check fails. The crucial thing I emphasize in interviews is that a JWT is NOT encrypted by default: the header and payload are only Base64url encoded, which is trivially reversible — anyone who intercepts the token can read all the claims. It is signed, not secret. If the payload contains sensitive data, use JWE — JSON Web Encryption — which encrypts the payload. For most auth tokens, the payload contains only non-sensitive identifiers like a user ID and roles, so signing is sufficient.

“Signed, not secret” — this one phrase shows you understand what a JWT actually guarantees.

Gotcha follow-up: Can you store a user’s password in a JWT payload? Absolutely not — the payload is only Base64-encoded, not encrypted. Anyone who intercepts the token can decode it with any Base64 decoder in seconds. Even “sensitive” claims like email or full name should be reconsidered — if the token flows over HTTP or gets logged by a proxy, those claims are exposed. Store the minimum necessary in the payload: user ID, roles, and expiry. Fetch sensitive profile data from the API on demand.

Q2: What is the difference between HS256 and RS256 signing algorithms for JWTs? Concept Check

One-line answer: HS256 uses a single shared secret that every verifier must know, making it risky in microservices; RS256 uses asymmetric keys so verifiers only need the public key, keeping the signing key safe.

Full answer: HS256 stands for HMAC-SHA256 — HMAC is Hash-based Message Authentication Code, and SHA-256 is the hash function. It is a symmetric algorithm: the same secret key is used both to sign the token and to verify the signature. This means every service that needs to validate JWTs must have a copy of that secret. In a monolith with one service, this is fine. In a microservices architecture, if you have twenty services all validating JWTs, all twenty need the secret — if any one of those services is compromised, an attacker can forge arbitrary JWTs for your entire system. RS256 stands for RSA-SHA256 and uses an asymmetric key pair: a private key that is held exclusively by the Authorization Server to sign tokens, and a corresponding public key that is distributed freely to all services for verification. A token signed with the private key can be verified with the public key, but you cannot forge a token using only the public key. The Authorization Server publishes its public keys at a standard JWKS endpoint — JSON Web Key Set, typically at /.well-known/jwks.json — and any service can fetch and cache those public keys. This is the correct architecture for microservices because no service other than the Authorization Server ever sees the private key; compromise of a downstream service does not allow token forgery.

The “twenty services all holding the secret” mental model is vivid and memorable for the interviewer.

Gotcha follow-up: How do you handle RS256 key rotation? The JWKS endpoint lists all currently valid public keys, each with a kid — key ID — field. JWTs include the kid in their header. When you rotate keys, add the new key pair to the JWKS endpoint before issuing tokens with the new private key. Services fetching the JWKS should cache the key set with a short TTL and re-fetch when they encounter a kid they do not recognize. The old key remains in JWKS until all tokens signed with it have expired. Never remove a key from JWKS while valid tokens signed with it are still in circulation.

Q3: What is the alg:none JWT attack and how do you defend against it? Concept Check

One-line answer: Early JWT libraries blindly trusted the algorithm in the token header — an attacker could set alg to “none” and submit a signature-less token that some libraries accepted as valid.

Full answer: The alg:none attack exploited a design mistake in early JWT library implementations. The vulnerability worked like this: an attacker took a legitimately signed JWT — say, one with valid user claims for a low-privilege user — stripped the signature, changed the header to specify “alg”: “none” — meaning no signature algorithm, no verification needed — and submitted this modified token to the server. Some early JWT libraries, following the JWT specification too literally, would parse the header, see the algorithm was “none”, skip signature verification (because “none” means no signature), and accept the token as valid. An attacker who did this could also modify the payload claims — for example, changing their role from “user” to “admin” — and the server would accept it. The defense is straightforward: your JWT validation code must never trust the algorithm specified in the incoming token’s header. Instead, explicitly configure your JWT library with the expected algorithm — for example, algorithm = RS256 — and reject any token whose header specifies a different algorithm. Any reputable modern JWT library allows you to specify the expected algorithm and will reject tokens with unexpected or “none” algorithms by default. Always check the library’s release notes and security advisories; this specific class of vulnerability has been patched in all major libraries, but only if you configure them correctly.

“Never trust the algorithm in the header” is the one-line takeaway — say it explicitly.

Gotcha follow-up: Are there other algorithm confusion attacks beyond alg:none? Yes — the RS256/HS256 confusion attack. Some buggy libraries accept either algorithm. An attacker takes a token signed with RS256, changes the header to HS256, and then signs the payload using the server’s public key (which is public knowledge) as the HMAC secret. A confused library that validates HS256 tokens using the RS256 public key would accept this forged token. Defense: same fix — explicitly configure the expected algorithm and reject anything else.

Q4: Walk me through the complete JWT validation checklist — what must you verify before trusting a JWT? Concept Check

One-line answer: Nine checks in order — format, algorithm, signature, expiry, not-before, issuer, audience, and optionally revocation and custom claims.

Full answer: When I receive a JWT in an Authorization header, I perform these checks in order. First, structural check: verify the token has exactly three dot-separated parts — a malformed token should be rejected before any parsing. Second, algorithm check: extract the alg field from the header and verify it matches the algorithm I expect — RS256, HS256, or whatever is configured — and reject immediately if it is “none” or anything unexpected. Third, signature verification: cryptographically verify the signature against the header and payload using the correct key — for RS256 this means fetching the public key from the JWKS endpoint and running the RSA verification. Fourth, expiry check: verify that the current timestamp is before exp — a Unix epoch timestamp — and reject if now is greater, meaning the token is expired. Fifth, not-before check: verify that now is greater than nbf — not before — if present; a token with nbf in the future is not yet valid. Sixth, issuer check: verify iss matches your trusted Authorization Server’s identifier — this prevents tokens from a different system being used against yours. Seventh, audience check: verify that aud — the audience claim — contains your service’s identifier; this prevents tokens issued for Service A from being accepted by Service B, a token replay attack. Eighth, optional revocation check: look up the jti — JWT ID — in a Redis revocation list to handle explicit token invalidation, for example when a user logs out. Ninth, custom claims: validate any application-specific claims required for the requested resource, such as checking that the user has the required role or the correct tenant ID.

Walking through all nine checks in order demonstrates the kind of methodical thinking that catches real vulnerabilities.

Gotcha follow-up: Which check do most developers skip that causes real vulnerabilities? The audience (aud) check. Most developers verify the signature and expiry but forget audience. Without it, a token issued by your Authorization Server for your mobile app can be replayed against your admin API, as long as both services share the same Authorization Server. The fix is a one-line configuration: set the expected audience in your JWT library. Many real-world token replay bugs come from this omission.

Q5: What are the four OAuth2 roles and what does each one do? Concept Check

One-line answer: Resource Owner is the user; Client is your application; Authorization Server issues tokens; Resource Server holds the protected data and validates tokens.

Full answer: OAuth2 is an authorization framework — it is about delegating access, not just authenticating. Understanding the four roles is essential to reasoning about any OAuth2 flow. The Resource Owner is the end user who owns the data and grants permission for it to be accessed — for example, a user who authorizes a third-party app to read their Google Drive files. The Client is the application that wants access to the resource on behalf of the user — this could be a web app, mobile app, or a backend service. The Client is not necessarily trusted; that is why OAuth2 requires explicit user consent. The Authorization Server is the trusted central server responsible for authenticating the user, obtaining their consent, and issuing tokens — examples include Keycloak, Auth0, Okta, and AWS Cognito. It exposes endpoints like /authorize for user login and consent, /token for exchanging codes or credentials for tokens, and /introspect or JWKS for token validation. The Resource Server is your API — the server that holds the actual protected resources. It validates access tokens on every request and returns data only if the token is valid and contains the required scopes or permissions. In a microservices architecture, you typically have one Authorization Server and many Resource Servers, all trusting the same Authorization Server’s public keys.

The phrase “delegating access, not just authenticating” is what distinguishes someone who truly understands OAuth2.

Gotcha follow-up: In a microservices setup, should each microservice validate JWTs independently or delegate to a central auth service? Each microservice should validate JWTs independently using the Authorization Server’s public key from the JWKS endpoint. Delegating to a central auth service creates a synchronous dependency — if the auth service is slow or down, every single API call in your entire system fails. Independent validation is stateless, fast, and resilient; the cost is fetching and caching the JWKS public keys, which is trivial.

Q6: Explain the Authorization Code flow with PKCE — why is PKCE necessary and how does it work? Design Scenario

One-line answer: PKCE — Proof Key for Code Exchange — prevents an intercepted authorization code from being exchanged for tokens by proving that the token requester is the same party that initiated the authorization.

Full answer: The Authorization Code flow is the standard OAuth2 flow for user-facing applications. Without PKCE, the flow is: the app redirects the user to the Authorization Server for login, the Authorization Server redirects back with a short-lived authorization code in the URL, and the app exchanges that code for tokens. The vulnerability is that the authorization code is in a URL and could be intercepted by a malicious app on the same device — especially on mobile where other apps can register the same URL scheme. PKCE adds a cryptographic proof to prevent the intercepted code from being usable. Here is how the PKCE flow works step by step. First, the client generates a random high-entropy string called the code_verifier — typically 43 to 128 characters, never sent over the network during the authorization step. Second, the client computes code_challenge equals SHA256(code_verifier), then Base64url-encodes the result. Third, the client redirects the user to the Authorization Server including the code_challenge and code_challenge_method=S256 as query parameters. The Authorization Server stores the code_challenge alongside the authorization session. Fourth, the user authenticates and consents, and the Authorization Server redirects back to the client’s callback URL with the authorization code. Fifth, the client sends the authorization code plus the original code_verifier — not the challenge — to the /token endpoint. The Authorization Server recomputes SHA256(code_verifier), compares it to the stored code_challenge, and only issues tokens if they match. An attacker who intercepted the authorization code does not have the code_verifier — they cannot compute the preimage of a SHA256 hash — so the code is useless to them.

Walking through each step in order, naming code_verifier and code_challenge specifically, signals you have actually implemented this.

Gotcha follow-up: Is PKCE only for public clients like mobile apps, or should confidential clients (backend servers) use it too? Modern best practice — as recommended by OAuth 2.1 — is that ALL clients should use PKCE, including confidential backends. PKCE provides defense-in-depth even when the client has a secret. The client_secret proves the client’s identity to the Authorization Server at token exchange time; PKCE additionally proves the token requester is the same party that started the authorization flow. There is no downside to using both.

Q7: How does the Client Credentials flow work and when should you use it? Design Scenario

One-line answer: Client Credentials is machine-to-machine OAuth2 — a service authenticates as itself using its client ID and secret to get an access token with no user involved.

Full answer: Many backend scenarios involve services calling other services without any user context — a nightly batch job archiving old records, a billing service querying an inventory service, or a message consumer calling an API to fulfill an order. These have no user to redirect to a login page. Client Credentials is the OAuth2 grant type designed for this. The calling service — the Client — sends a POST request directly to the Authorization Server’s /token endpoint with grant_type=client_credentials, its client_id, its client_secret, and optionally the scope of access requested. The Authorization Server authenticates the service by verifying the client_id and client_secret, then issues an access token representing the service’s own identity and permissions — not any user’s. There is no refresh token in Client Credentials because the service can always request a new access token directly. The access token should be cached until near its expiry to avoid requesting a new token on every API call — a common optimization is to cache the token and refresh it when it has less than 30 seconds of remaining validity. Security requirements: the client_secret must never be hardcoded in source code or environment variables in plaintext — store it in a secrets manager like HashiCorp Vault, AWS Secrets Manager, or Kubernetes Secrets, and inject it at runtime. Rotate secrets regularly using the secrets manager’s rotation feature.

Mentioning token caching and “never hardcode the secret” shows production awareness beyond just the happy path.

Gotcha follow-up: How do you scope down permissions for a Client Credentials token to follow least privilege? Define specific OAuth2 scopes for each service-to-service interaction — for example, inventory:read for a service that only needs to query inventory, versus inventory:write for a service that can also update it. The calling service requests only the scopes it needs. The Resource Server validates that the token’s scope includes what is required for the specific endpoint. This way, if a service is compromised, the blast radius is limited to only the permissions that service was granted.

Q8: What is the difference between an OIDC ID Token and an OAuth2 Access Token? Concept Check

One-line answer: The ID Token tells your application WHO the user is; the Access Token tells your API what the user is ALLOWED to do — never send the ID Token to your Resource Server.

Full answer: OpenID Connect, abbreviated OIDC, is a thin identity layer built on top of OAuth2. OAuth2 defines how to authorize access to resources but says nothing about who the user is — it only issues tokens for accessing APIs. OIDC adds the identity piece. When a user logs in via an OIDC flow, the Authorization Server issues three items: an ID Token, an Access Token, and optionally a Refresh Token. The ID Token is a JWT specifically for the client application — your frontend or backend-for-frontend. It contains claims about the user’s identity: sub is the unique user identifier, email, name, picture, and whatever other profile attributes the user consented to share. Your application reads the ID Token to establish the user’s identity — who just logged in. The Access Token is for the Resource Server — your API. It contains scopes representing what the user authorized the client to do on their behalf, and it may contain roles or permissions. Your API validates the Access Token on every request to decide whether to serve the data. The critical rule: never send the ID Token to your API, and never use the Access Token as proof of who the user is in your frontend. The Refresh Token is a long-lived token stored securely — in a server-side session or an httpOnly, Secure, SameSite cookie — used to silently obtain new Access Tokens without requiring the user to log in again when the Access Token expires. Never store Refresh Tokens in localStorage or sessionStorage — they are accessible to any JavaScript on your page and are stolen in XSS attacks.

The “never send the ID Token to your API” rule is broken constantly in real codebases — naming it is a strong signal.

Gotcha follow-up: What is the userinfo endpoint and when would you use it instead of reading ID Token claims? The userinfo endpoint is an OAuth2/OIDC protected endpoint at the Authorization Server that returns the same user profile claims as the ID Token, but it always returns the current, up-to-date data. The ID Token’s claims are baked in at issuance time — if the user updates their email after the token is issued, the ID Token still shows the old email until it expires. The userinfo endpoint is useful when you need fresh, authoritative user profile data and cannot wait for token expiry. The tradeoff is an extra HTTP call. For most applications, the ID Token claims are sufficient; use userinfo for profile displays where currency matters.

Q9: How does the Spring Security filter chain work and where does JWT validation fit in? Design Scenario

One-line answer: Spring Security is an ordered chain of servlet filters; JWT validation must be inserted before the standard authentication filters using addFilterBefore.

Full answer: Spring Security is implemented as a javax.servlet.Filter chain — a series of filter objects that each HTTP request passes through in order, where each filter can inspect, modify, or block the request before passing it to the next filter. Understanding the order is critical for configuring custom authentication. SecurityContextPersistenceFilter runs first and restores the security context — the object holding the current authenticated user — from the session or from a token on each request. Next come the authentication filters: UsernamePasswordAuthenticationFilter handles form-based login, BasicAuthenticationFilter handles HTTP Basic authentication with Base64-encoded credentials, and BearerTokenAuthenticationFilter handles Bearer token validation for OAuth2 and JWT. These filters attempt to authenticate the request and, if successful, populate the SecurityContext with an Authentication object containing the user’s identity and authorities. ExceptionTranslationFilter catches two specific exceptions: AuthenticationException, which maps to a 401 Unauthorized response when the request is not authenticated, and AccessDeniedException, which maps to a 403 Forbidden response when the request is authenticated but lacks permission. Finally, AuthorizationFilter — formerly FilterSecurityInterceptor — checks the authenticated user’s authorities against the configured access rules for the requested URL. To add a custom JWT filter — for example, when using a non-standard token format — extend OncePerRequestFilter, parse and validate the JWT, construct a UsernamePasswordAuthenticationToken with the user’s ID and authorities, and set it on SecurityContextHolder. Register it with http.addFilterBefore(myJwtFilter, UsernamePasswordAuthenticationFilter.class) in your SecurityFilterChain configuration.

Naming ExceptionTranslationFilter and explaining 401 vs 403 mapping within the chain shows depth beyond the happy path.

Gotcha follow-up: What is the difference between antMatchers and authorizeHttpRequests in Spring Security 6? antMatchers was deprecated in Spring Security 6 and replaced by requestMatchers. More importantly, the underlying authorization model changed: the old FilterSecurityInterceptor used a voter-based model; the new AuthorizationFilter uses a cleaner AuthorizationManager interface. The configuration syntax is similar but use authorizeHttpRequests() with requestMatchers() in Spring Security 6+, not authorizeRequests() with antMatchers(). The old methods still work but are deprecated and will be removed.

Q10: How does @PreAuthorize work and what SpEL expressions are most useful? Concept Check

One-line answer: @PreAuthorize uses Spring Expression Language evaluated before method execution to enforce authorization — the most powerful expressions check ownership, not just roles.

Full answer: @PreAuthorize is a method security annotation that evaluates a Spring Expression Language expression before the annotated method runs. If the expression evaluates to false, Spring throws an AccessDeniedException before the method body executes. Spring Expression Language, or SpEL, is a powerful expression syntax that provides access to the security context and method parameters. hasRole(‘ADMIN’) checks whether the authenticated user has the ADMIN role — Spring prepends the ROLE_ prefix automatically, so it looks for ROLE_ADMIN in the user’s granted authorities. hasAuthority(‘SCOPE_read’) checks for an exact authority string without any prefix — this is used for OAuth2 scopes where the authority is SCOPE_read. The most important and often overlooked expression is ownership validation: #userId == authentication.principal.id compares a method parameter named userId — the hash symbol accesses method parameters by name — to the authenticated user’s ID from the security context. Without this, an authenticated user could access any other user’s data simply by changing the ID in the request. authentication.principal gives access to the full UserDetails or JWT principal object. For complex permission logic that cannot be expressed inline, delegate to a Spring bean: @securityService.canAccess(#resourceId, authentication) calls the canAccess method on a bean named securityService, passing the resource ID and authentication object. @PostAuthorize runs after the method and can filter or block based on the return value — useful for checking that the returned object belongs to the current user. Enable everything with @EnableMethodSecurity on a @Configuration class.

The #userId == authentication.principal.id example is the one that prevents BOLA vulnerabilities — always mention it.

Gotcha follow-up: How do you unit test @PreAuthorize without starting the full Spring context? Use Spring Security Test’s @WithMockUser annotation or SecurityMockMvcRequestPostProcessors.user() to populate the SecurityContext with a test user. For @PreAuthorize with custom expressions, use @WithSecurityContext to build a fully custom authentication object. You can test method security without MockMvc by directly calling the secured method from a test that has @SpringBootTest and the Spring context loaded — the AOP proxy that evaluates @PreAuthorize is active in the full context.

Q11: Why is bcrypt not always sufficient for password hashing, and when should you use Argon2id? Tradeoff Question

One-line answer: bcrypt is solid but has a 72-character input limit and is not memory-hard; Argon2id is the current best practice because its memory requirement makes GPU and ASIC attacks impractical.

Full answer: bcrypt has been a respected password hashing function since 1999 and remains far better than any general-purpose hash like SHA-256 or MD5 for passwords. Its key feature is the work factor: a cost parameter where increasing the value by one doubles the computation time, making it adaptive as hardware gets faster. The standard recommendation is a cost of 12, which takes roughly 300 milliseconds — fast enough for user login but expensive enough to make brute force searches impractical. However, bcrypt has two weaknesses. First, it silently truncates input at 72 characters — a password of 73 characters produces the same hash as the first 72 characters of that password, which could be exploited if users have very long passwords and an attacker knows this. Second, and more importantly, bcrypt is not memory-hard: attackers can build custom ASIC hardware or use GPUs to compute billions of bcrypt hashes per second in parallel, because the algorithm requires minimal RAM. Argon2id won the 2015 Password Hashing Competition and is the current recommendation from NIST and OWASP. It is memory-hard: you configure a memory-cost parameter specifying how much RAM each hash computation requires — typically 64 MB or more. Because GPU and ASIC hardware has limited RAM per compute unit, parallelizing Argon2id attacks requires proportionally more RAM per parallel unit, making massive GPU farms economically impractical. Argon2id has no input length limit. The id variant balances resistance to both side-channel attacks (Argon2i) and GPU attacks (Argon2d). Spring Security supports Argon2PasswordEncoder — use it for all new systems. If migrating from bcrypt, hash existing passwords with Argon2 on next login via a progressive re-hash strategy.

Explaining memory-hard in terms of “limited RAM per GPU compute unit” makes the concept concrete.

Gotcha follow-up: What is the pre-hashing trick when using bcrypt with long passwords? Because bcrypt truncates at 72 characters, a common mitigation is to pre-hash the password with SHA-256 before passing it to bcrypt: bcrypt(base64(sha256(password))). The SHA-256 output is always 32 bytes, well within bcrypt’s limit, and the Base64 encoding avoids null bytes that some bcrypt implementations handle poorly. However, the cleaner solution is to just use Argon2id, which has no input limit and is more secure overall. Pre-hashing is a legacy mitigation for systems already committed to bcrypt.

Q12: Explain the TLS handshake in simple terms — what problem does it solve and how? Concept Check

One-line answer: TLS establishes a shared encryption key over an untrusted network so that subsequent communication is encrypted and the server’s identity is verified.

Full answer: TLS — Transport Layer Security — solves two problems simultaneously: it verifies that you are really talking to the server you think you are (authentication), and it establishes a shared encryption key so all subsequent data is unreadable to anyone intercepting the connection (confidentiality). In the simplified TLS 1.3 handshake, the client sends a Client Hello containing the TLS version it supports, a list of cipher suites — algorithm combinations it understands — and its own key share, which is a public value for the key exchange. The server responds with a Server Hello picking the cipher suite, its own key share, and crucially its certificate. The certificate is a signed document containing the server’s public key and identity information — it is signed by a Certificate Authority, a trusted third party like Let’s Encrypt or DigiCert. The client verifies the certificate: it checks the CA’s signature, confirms the certificate has not expired, checks that it has not been revoked, and verifies that the domain name in the certificate matches the hostname you are connecting to. This prevents a man-in-the-middle from substituting their own certificate — they cannot forge a certificate signed by a trusted CA without controlling the CA. Both sides now compute the same session key from their key shares using ECDHE — Elliptic Curve Diffie-Hellman Ephemeral. The key insight of Diffie-Hellman is that both parties can compute the same shared secret without ever sending that secret over the wire, even though the messages exchanged are public. All data after the handshake is encrypted with this session key. The Ephemeral part means a new key pair is generated for every session — this provides Perfect Forward Secrecy, meaning that even if the server’s long-term private key is compromised in the future, past session keys cannot be derived and old captured traffic cannot be decrypted.

Perfect Forward Secrecy is the detail that separates a surface-level TLS answer from a deep one.

Gotcha follow-up: What is certificate pinning and when should you use it? Certificate pinning hardcodes the expected certificate or public key hash in the client application — typically a mobile app — so the client rejects any certificate that does not match, even if it is signed by a trusted CA. This prevents attacks where an attacker tricks the OS into trusting a rogue CA and uses it to issue a legitimate-looking certificate for your domain. The downside is operational: when you rotate your certificate, you must update all pinned clients simultaneously. Use pinning for mobile apps handling highly sensitive data — banking, healthcare — but be prepared for the operational overhead of managed key rotation.

Q13: What is the difference between CORS and CSRF, and how should each be handled in a Spring REST API? Concept Check

One-line answer: CORS prevents unauthorized cross-origin JavaScript requests; CSRF prevents forged same-user form submissions using session cookies — and CSRF does not apply to stateless REST APIs using Bearer tokens.

Full answer: CORS — Cross-Origin Resource Sharing — is a browser security policy that blocks JavaScript on one origin from making HTTP requests to a different origin. An origin is the combination of scheme, host, and port — so https://app.example.com and https://api.example.com are different origins. Without CORS headers, a browser will block the response from a cross-origin request even if the request was sent. Your API opts in to trusted origins by including Access-Control-Allow-Origin in its responses. In Spring, configure CORS via WebMvcConfigurer.addCorsMappings() or the @CrossOrigin annotation — specify exactly which origins, methods, and headers are allowed, and never use wildcard * in production for authenticated APIs. CSRF — Cross-Site Request Forgery — is a completely different attack: a malicious website tricks a user’s browser into making a state-changing request — like submitting a transfer form — to your API using the user’s existing session cookie. The browser automatically includes the session cookie with the request because the cookie’s domain matches your API, even though the request originated from the malicious site. Defense is the CSRF token: a secret value embedded in forms and verified by the server — a cross-origin page cannot read it due to the same-origin policy. The key insight for REST API design: if your API is stateless — meaning it authenticates via a Bearer token in the Authorization header, not a session cookie — CSRF does not apply. Cross-origin JavaScript cannot read the Authorization header value from another page, and cookies are not involved. Disable CSRF protection in Spring Security for REST APIs with http.csrf().disable() — this is correct, not a security hole, when you are using Bearer tokens.

Saying “disabling CSRF for Bearer token APIs is correct” confidently shows you understand the threat model, not just the rule.

Gotcha follow-up: If you are using a httpOnly session cookie for auth instead of a Bearer token in a header, does CSRF then apply? Yes, absolutely. If your API authenticates via a session cookie — even an httpOnly one — CSRF applies because the browser automatically sends cookies with cross-origin requests. httpOnly prevents JavaScript from reading the cookie, but does not prevent the browser from sending it with a forged form submission. In this case, use Spring Security’s built-in CSRF protection with double-submit cookie or synchronizer token patterns.

Q14: What does the SameSite cookie attribute do and which value should you use for auth cookies? Concept Check

One-line answer: SameSite controls whether browsers send a cookie with cross-site requests; use SameSite=Lax with HttpOnly=true and Secure=true for auth cookies to balance security and usability.

Full answer: The SameSite attribute was introduced as a browser-level CSRF mitigation. It tells the browser under what circumstances to include the cookie when a request is triggered from a different site. SameSite=Strict means the cookie is only sent when the origin of the request exactly matches the cookie’s domain — if a user clicks a link in an email or a third-party website to your site, the cookie is NOT sent on that first request, which means the user appears unauthenticated and may get redirected to login even though they already have a valid session. This is the most secure option but creates a jarring user experience for link-following scenarios. SameSite=Lax is the browser default in modern browsers when no SameSite attribute is specified. The cookie is sent with same-site requests and also with top-level navigations like clicking a link — so a user clicking a link from Google to your site will have their session cookie sent, enabling seamless single-click access. However, Lax blocks the cookie for cross-site subresource requests, forms, and XHR — the attack vectors CSRF exploits. This makes Lax the right balance for most auth cookies. SameSite=None means the cookie is sent with all cross-origin requests, which is required for third-party integrations — like an embedded widget served from your domain — but requires the Secure attribute, meaning it only works over HTTPS. For a production auth cookie, combine HttpOnly=true — the JavaScript document.cookie API cannot read the cookie, mitigating XSS-based session theft — with Secure=true — the cookie is only sent over HTTPS — with SameSite=Lax — CSRF protection without breaking link navigation. Set this explicitly; do not rely on browser defaults.

Explaining WHY Strict breaks link-following is what most people miss — it shows you tested this, not just read about it.

Gotcha follow-up: Can SameSite=Lax fully replace CSRF tokens? For most modern browsers, SameSite=Lax provides sufficient CSRF protection and CSRF tokens may be redundant. However, defense-in-depth means keeping both: older browsers do not enforce SameSite, and some mobile browsers have inconsistent SameSite handling. OWASP still recommends CSRF tokens as the primary defense, with SameSite as an additional layer. The effort to add both is low; the risk reduction is meaningful for cross-browser compatibility.

Q15: Walk me through the OWASP API Security Top 10 and give an example of BOLA. Design Scenario

One-line answer: The OWASP API Top 10 are the most critical API vulnerabilities; BOLA — Broken Object Level Authorization — is consistently number one and means failing to verify that the requesting user owns the object they are accessing.

Full answer: The OWASP API Security Top 10 is the authoritative reference for API vulnerabilities that are commonly exploited in the real world. Let me walk through the most critical ones. BOLA — Broken Object Level Authorization — is number one and the most common. It means your API accepts requests for specific objects by ID but does not verify that the authenticated user is authorized to access that specific object. For example, GET /api/orders/42 returns the order with ID 42 — but does the code check that order.userId == currentUser.id? If not, any authenticated user can access any order by guessing IDs. The fix is always: after fetching the resource, assert ownership before returning it. Broken Authentication — number two — includes weak JWT validation, short-lived tokens without proper expiry, and exposed secrets in logs. Broken Object Property Level Authorization — number three — is mass assignment: if your endpoint does user.setRole(dto.getRole()) directly from the request body, an attacker can promote themselves to admin. Use explicit DTOs with only the fields users are allowed to set. Unrestricted Resource Consumption — number four — includes missing rate limiting, unbounded pagination (allowing page size of one million), and unlimited file uploads. BFLA — Broken Function Level Authorization — number five — means ordinary users can call admin-only functions because you check authentication but not authorization for sensitive operations like DELETE /users or GET /admin/export. SSRF — Server-Side Request Forgery — number seven — occurs when your API fetches a URL supplied by the user and an attacker points it at internal services like http://169.254.169.254/latest/meta-data/ to steal cloud credentials. Security Misconfiguration — number eight — includes debug endpoints left exposed in production, default credentials unchanged, and verbose error messages that reveal stack traces and internal paths to attackers.

Leading with BOLA and giving a concrete code example is what separates a theoretical answer from a practical one.

Gotcha follow-up: How do you prevent BOLA at scale in a codebase with hundreds of endpoints? Three approaches: first, create a base repository or service layer that always appends an ownership filter to queries — for example, findByIdAndUserId(id, currentUserId) — so developers cannot accidentally call findById(id) alone. Second, use integration tests that explicitly test accessing resources as a different user — these tests catch BOLA regressions in CI. Third, use a policy engine like OPA — Open Policy Agent — or Spring Security’s @PostAuthorize annotations that evaluate ownership rules consistently without requiring every developer to remember them inline.

Common Mistakes — Section 5:

  • Not validating the aud (audience) claim → tokens issued for your mobile app get accepted by your admin API. Always configure expected audience in your JWT library.
  • Storing Refresh Tokens in localStorage → accessible to any JavaScript on the page; stolen via XSS. Store in httpOnly, Secure, SameSite cookies.
  • Disabling CSRF for APIs using session cookies → CSRF is only safe to disable when using Bearer tokens in Authorization headers. If you use cookies for auth, keep CSRF protection enabled.
  • Using HS256 in microservices → all services need the shared secret; one compromised service lets attackers forge tokens for the entire system. Use RS256 with JWKS.
  • Mass assignment — accepting role from request body → use explicit DTOs with only fields users are permitted to set; never auto-map request bodies directly to JPA entities with sensitive fields.

Quick Revision: JWT checklist — structure, algorithm, signature, exp, nbf, iss, aud, jti, custom claims. BOLA = no ownership check. RS256 = asymmetric, safe for microservices. CSRF = irrelevant for Bearer token APIs.

Reading guide: Five synthesizing questions that test your ability to compare related concepts and articulate WHY the distinctions matter in practice. These are likely to come up as follow-up questions after deeper dives.

Q1: What is the difference between safe and idempotent HTTP methods, and why does this matter in API design? Concept Check

One-line answer: Safe means the method has no side effects; idempotent means repeating the request N times has the same effect as doing it once — these two properties drive retry logic and caching decisions.

Full answer: These two properties are foundational to building reliable distributed HTTP APIs because networks are unreliable — requests can fail mid-flight, and clients need to know whether it is safe to retry. Safe methods are those that do not modify server state: GET, HEAD, and OPTIONS are safe. You can call GET a hundred times and nothing changes. Because they are safe, responses to GET requests can be cached by proxies and browsers — a safe method is a read, and reads are cacheable. Idempotent methods can be called multiple times and the server state ends up the same as if you called it once: GET, HEAD, OPTIONS, PUT, and DELETE are all idempotent. PUT with the same full resource representation applied twice results in the same state as applying it once. DELETE applied twice — the second call returns 404, but the resource is still deleted, so the outcome is the same. POST is neither safe nor idempotent — calling POST /orders twice creates two orders. PATCH is interesting: it is not guaranteed to be idempotent. A PATCH that says “set status to SHIPPED” is idempotent. A PATCH that says “increment retryCount by 1” is not — applying it twice increments twice. The practical consequence: your HTTP client or API gateway can safely auto-retry idempotent methods on timeout or network error; it must not auto-retry non-idempotent methods without application-level deduplication logic. This is why idempotency keys — a client-generated unique ID per operation sent as a header — are used with POST for payment APIs.

The idempotency key pattern for POST is the real-world application that elevates this answer from theoretical to practical.

Gotcha follow-up: Is DELETE truly idempotent if the second call returns 404? Yes — idempotency is about the server state, not the response code. After both the first and second DELETE calls, the resource is absent. The state is the same. The 404 on the second call is technically correct behavior and does not violate idempotency. Some APIs return 204 on both calls to make client retry logic simpler — both are valid choices.

Q2: Compare the OAuth2 grant types — when would you use Authorization Code, Client Credentials, and Device Code? Tradeoff Question

One-line answer: Authorization Code for user-facing apps, Client Credentials for machine-to-machine, Device Code for input-constrained devices like smart TVs — pick based on whether there is a user and whether there is a browser.

Full answer: The OAuth2 grant type answers one question: how does the client prove it is allowed to get an access token? Authorization Code with PKCE is the right choice when a human user must consent and there is a browser available for the redirect flow. The user is sent to the Authorization Server, authenticates there — keeping their credentials off your app — consents to the requested scopes, and is redirected back. PKCE is always required for public clients — mobile apps and SPAs — and is best practice for all clients. Client Credentials is the right choice when there is no user — a backend service, cron job, or microservice calling another microservice. The service authenticates using its own client ID and secret, gets a token representing itself, and calls the API. No redirect, no user interaction, no refresh token needed. Device Code is for input-constrained devices — smart TVs, game consoles, printers, IoT devices — where typing a complex URL is impractical. The device gets a short code, displays it on screen with a URL, and polls the Authorization Server waiting for the user to visit the URL on their phone or laptop and enter the code. Implicit grant — where the token is returned directly in the redirect URL — is fully deprecated; never use it because the token is visible in browser history and referrer headers. Resource Owner Password Credentials — where the user gives their username and password directly to the client — is also deprecated except for first-party legacy migration scenarios, because it violates the OAuth2 principle of the client never seeing user credentials.

Calling out both deprecated flows — Implicit and ROPC — shows you know the security history of the protocol.

Gotcha follow-up: When should you use Authorization Code with a backend-for-frontend versus a pure SPA with tokens in memory? In a pure SPA, the access token and refresh token live in browser memory or localStorage — accessible to any script on the page, vulnerable to XSS. The Backend-for-Frontend pattern — BFF — moves token handling to a server: the BFF completes the Authorization Code flow, stores the refresh token in an httpOnly, Secure, SameSite cookie, and proxies API calls to Resource Servers using the access token. The SPA never sees a token; it just sends session cookies to the BFF. This is the recommended pattern for SPAs handling sensitive data because it eliminates the XSS token theft risk.

Q3: What are the Kafka delivery semantics — at-most-once, at-least-once, and exactly-once — and what configuration achieves each? Concept Check

One-line answer: At-most-once allows message loss; at-least-once allows duplicates; exactly-once prevents both — and exactly-once requires idempotent producers, transactional.id, and read_committed consumers working together.

Full answer: Delivery semantics describe what guarantee a messaging system makes about message delivery in the presence of failures. At-most-once means a message is delivered zero or one time — it will never be delivered twice, but it might be dropped. This happens when the consumer uses Kafka’s auto-commit and commits the offset before processing finishes — if the consumer crashes after committing but before processing, the message is lost. This is the default behavior with enable.auto.commit=true. Use at-most-once only for metrics, logging, or real-time data where occasional loss is acceptable. At-least-once means every message is eventually delivered, but may be delivered more than once if a failure causes a retry. This is achieved by committing the offset only after successfully processing the message — manual commit with ack.acknowledge() in Spring Kafka. The consumer must be idempotent — designed to handle duplicate messages safely, for example by using a unique message ID to detect and discard duplicates. This is the most common production configuration because it is relatively simple and most business logic can be made idempotent. Exactly-once means every message is delivered and processed precisely once, with no loss and no duplicates. This requires three things working together: an idempotent producer — enable.idempotence=true — which deduplicates retries using a sequence number so the broker ignores duplicate sends; a transactional producer — configured with transactional.id — which can publish to multiple partitions and topics atomically; and consumers with isolation.level=read_committed which only see messages from committed transactions and skip aborted ones. Exactly-once is significantly more complex and has throughput overhead — use it only when the cost of a duplicate is high, such as in financial transactions or charge operations.

“The three things working together” framing makes it clear you understand this is a system-level property, not a single setting.

Gotcha follow-up: Does enable.idempotence=true on the producer alone give you exactly-once semantics? No — idempotent producer gives you exactly-once at the producer-to-broker level for a single partition. It deduplicates retries within a producer session, preventing duplicate messages from appearing in Kafka due to network retries. But it does not prevent your consumer from processing the same message twice if the consumer crashes after processing but before committing its offset. True end-to-end exactly-once requires the full transactional producer and read_committed consumer combination.

Q4: What are the Redis eviction policies and when would you choose allkeys-lru versus allkeys-lfu? Tradeoff Question

One-line answer: allkeys-lru evicts the least recently used key — best for general caches; allkeys-lfu evicts the least frequently used — better when access patterns are heavily skewed with a stable hot set.

Full answer: I covered all six eviction policies in depth in Section 4 Q6, so here I will focus specifically on the lru vs lfu decision, which comes up often because both are valid general-purpose policies and the choice depends entirely on your access pattern. LRU — Least Recently Used — tracks when each key was last accessed and evicts the one that has not been accessed in the longest time. LRU assumes that recently-used data is more likely to be needed again soon — a recency bias. This is an excellent assumption for most web application caches: a user who just viewed product page 42 is likely to view it again in the next few seconds, and a product page not viewed in a week is unlikely to be viewed in the next few seconds. The weakness of LRU is scan resistance: if a batch job reads through a large dataset once, it pollutes the LRU list with one-time-use keys, evicting frequently-accessed hot keys that had not been used in the last few milliseconds. LFU — Least Frequently Used — tracks how many times each key has been accessed and evicts the key with the lowest access count. LFU is better when your cache has a stable, long-term hot set — a small number of extremely popular items that receive the vast majority of traffic over a long period. Think product recommendations for the top 100 bestsellers: they receive millions of requests per day but might have gaps of a few seconds between requests; LRU might evict them during those gaps, while LFU would never evict them because their lifetime frequency is enormous. The practical advice: start with allkeys-lru because it is well-understood and covers most cases. If you see hot keys being unexpectedly evicted in Redis metrics — use the object freq and object idletime commands to diagnose — consider switching to allkeys-lfu.

The batch scan contamination scenario for LRU is the concrete failure case that makes the LFU answer make sense.

Gotcha follow-up: How does Redis implement LRU — does it actually maintain a sorted access log?** Redis does not maintain a true LRU list because the overhead would be prohibitive. Instead, Redis uses an approximated LRU: when eviction is needed, it randomly samples a configurable number of keys (default 5, configurable via maxmemory-samples) and evicts the one with the oldest last-access time among the samples. Increasing maxmemory-samples improves accuracy at the cost of more CPU per eviction. Redis 4.0 added a true LFU implementation using a logarithmic counter to approximate access frequency with just a few bytes per key.

Q5: Compare the Resilience4j patterns — circuit breaker, retry, bulkhead, and rate limiter — what problem does each solve and when would you use them together? Design Scenario

One-line answer: Circuit breaker stops cascade failures; retry handles transient failures; bulkhead isolates thread pools to prevent one slow dependency from exhausting all threads; rate limiter prevents overloading downstream services.

Full answer: These four patterns are the core toolkit for making a microservice resilient against the failures of its dependencies. Circuit Breaker solves the cascade failure problem: when a downstream service is slow or failing, every call to it ties up a thread for the full timeout duration. Without a circuit breaker, all your threads fill up waiting for the failing service, and your service becomes slow or unresponsive too — the failure propagates upstream. The circuit breaker monitors the failure rate of calls to the dependency. When failures exceed a threshold — say 50% of the last 100 calls — the circuit trips to OPEN state and immediately rejects all calls with a fallback response, without attempting the real call. After a configurable wait, it enters HALF-OPEN state and allows a few test calls through. If they succeed, the circuit closes and normal operation resumes. Retry solves transient network failures — a packet lost, a brief connection hiccup, a momentary overload. With retry, a failed call is automatically retried a configurable number of times with exponential backoff — waiting 100ms, then 200ms, then 400ms — with jitter added to prevent synchronized retry storms. Only apply retry to idempotent operations; retrying a non-idempotent operation can cause duplicates. Bulkhead solves thread pool exhaustion: it gives each downstream dependency its own bounded thread pool. If the payment service is slow and fills its bulkhead pool, calls to the inventory service continue on their separate pool. Without bulkhead, one slow dependency fills the global thread pool and all other calls fail. Rate Limiter protects downstream services from being overwhelmed by limiting how many calls per second your service makes to them — a courtesy to your dependency. In a resilient service call, stack them in order: Rate Limiter → Circuit Breaker → Retry → Bulkhead → call.

The “stack them in order” sentence is the synthesis that shows you have used these in production together.

Gotcha follow-up: What is the difference between a circuit breaker and a timeout? A timeout prevents a single call from waiting forever — it bounds the latency of one request. A circuit breaker aggregates the failure pattern of many calls over time and proactively stops sending requests when the failure rate is high. You need both: a timeout prevents individual requests from hanging, and the circuit breaker uses timeout-induced failures as signal to trip open. Without a circuit breaker, you still exhaust threads even with timeouts, because failures are frequent enough that new requests keep arriving and occupying thread slots for the full timeout duration.

Common Mistakes — Section 6:

  • Assuming PATCH is always idempotent → only “set to value” patches are idempotent; “increment by N” patches are not. Document your PATCH semantics explicitly.
  • Using Implicit OAuth2 grant → tokens appear in URL fragments visible in browser history and referrer headers. Always use Authorization Code with PKCE.
  • Thinking idempotent producer alone gives exactly-once → it prevents duplicate sends to the broker but not duplicate consumer processing. Full exactly-once requires transactional producer and read_committed isolation.
  • Defaulting to allkeys-lru without considering traffic patterns → LRU is correct for most cases, but if you observe hot keys being evicted, profile with object freq before assuming it is a memory sizing issue.
  • Not combining Resilience4j patterns → circuit breaker without retry means transient failures are not recovered; retry without circuit breaker means retries pile up during an outage. Layer them.

Quick Revision: Safe = no side effects (GET). Idempotent = repeat N times = same result (GET, PUT, DELETE). POST is neither. Exactly-once = idempotent producer + transactional.id + read_committed.

Section 7: Common Interview Traps

Reading guide: 20 traps that experienced interviewers specifically use to catch candidates who have surface-level knowledge. Each has a mechanism — why the wrong answer fails — and a fix. Know all 20 before your interview.

Trap 1: Confusing 401 and 403 HTTP status codes

The trap: saying “unauthorized” when you mean forbidden, or using 401 when authentication succeeds but authorization fails. A 401 Unauthorized response means the request is not authenticated — the server does not know who you are, and it includes a WWW-Authenticate header telling the client how to authenticate. The name “Unauthorized” is a historical misnomer; it really means “Unauthenticated.” A 403 Forbidden means the server knows exactly who you are — you are authenticated — but you do not have permission to perform this action. The fix: return 401 when the request has no valid credentials or the token is expired; return 403 when the user is authenticated but lacks the required role or ownership. Some APIs return 404 instead of 403 for sensitive resources to avoid disclosing that the resource exists — this is a valid security pattern called security through obscurity, though true authorization is still required.

Trap 2: Assuming PATCH is always idempotent

The trap: blanket-stating “PATCH is idempotent” because it is a partial update. Idempotency depends on the semantics of the patch, not the method itself. A PATCH that says “set status to ACTIVE” is idempotent — applying it ten times results in status ACTIVE every time. A PATCH that says “increment retryCount by 1” is not idempotent — applying it ten times increments the counter ten times. The fix: never assume PATCH is idempotent. In API documentation and client code, explicitly note whether a PATCH endpoint is idempotent. If retry logic is applied to PATCH requests, ensure the operation is set-based, not increment-based, or use an idempotency key header.

Trap 3: Using OFFSET pagination on large datasets

The trap: implementing pagination as SELECT * FROM orders LIMIT 20 OFFSET 1000000. OFFSET tells the database to scan and discard one million rows before returning the next twenty — this is an O(N) operation per page, so reading page fifty thousand is fifty thousand times slower than page one. At scale, deep pagination causes query timeouts and degrades overall database performance. The fix: use keyset pagination, also called cursor-based pagination — WHERE id > lastSeenId ORDER BY id ASC LIMIT 20. The WHERE clause uses the index on id directly, skipping to the last seen position in O(log N) time regardless of how deep into the dataset you are. Expose the last seen ID as an opaque cursor in the API response. The downside: you cannot jump to arbitrary pages, only forward or backward from the current cursor, which is acceptable for most API use cases.

Trap 4: Kafka auto-commit creating at-most-once risk

The trap: relying on Kafka’s default enable.auto.commit=true and believing you have at-least-once delivery. With auto-commit, Kafka commits the offset on a fixed schedule — every five seconds by default — regardless of whether your application has actually processed the messages. If your consumer processes a message, and the auto-commit fires shortly after, and then the consumer crashes before processing the next batch, those messages are lost — the offset was committed, so they will never be redelivered. The fix: disable auto-commit with enable.auto.commit=false and commit manually after your processing logic completes. In Spring Kafka with @KafkaListener, set AckMode.MANUAL and call acknowledgment.acknowledge() only after your business logic succeeds. This gives you at-least-once semantics — the offset is only committed after confirmed processing.

Trap 5: Kafka consumer group rebalance storms

The trap: not understanding that frequent consumer group rebalances can cause significant lag and throughput degradation. A rebalance is triggered whenever a consumer joins or leaves the group, which includes routine events like deployments, restarts, and crashes. During a rebalance, all consumers in the group stop processing — a stop-the-world event — while partitions are redistributed. If rebalances happen too frequently — for example, because a consumer’s processing takes longer than max.poll.interval.ms, causing Kafka to think the consumer is dead and triggering a rebalance — your consumer group can spend more time rebalancing than processing. The fix: increase max.poll.interval.ms to a value comfortably above your maximum processing time; reduce max.poll.records so each poll fetches fewer messages that can be processed within the interval; and use the CooperativeStickyAssignor partition assignment strategy instead of the default RangeAssignor — cooperative rebalancing allows only the partitions that need to move to be reassigned, rather than revoking all partitions from all consumers and redistributing from scratch.

Trap 6: Using Redis KEYS * in production

The trap: using the KEYS pattern command to find all keys matching a pattern in a production Redis instance. KEYS is an O(N) command where N is the total number of keys in the database. Because Redis uses a single-threaded event loop for command processing, KEYS blocks all other commands — including reads and writes — for the entire duration of the scan. On a Redis instance with millions of keys, this can cause seconds of latency for all other clients, effectively a self-inflicted denial of service. The fix: always use SCAN with a cursor instead. SCAN returns a small batch of results per call and does not block other operations between calls. Iterate with successive SCAN calls until the cursor returns to zero. The tradeoff is that SCAN is not atomic — keys may be added or removed during the scan — but for tasks like monitoring, debugging, or cleanup this is acceptable.

Trap 7: Assuming JWTs can be revoked like session tokens

The trap: treating JWT-based authentication as if tokens can be instantly invalidated like deleting a session from a session store. JWTs are stateless and self-contained — once issued, a token is valid until its exp timestamp passes, regardless of what happens server-side. You cannot “delete” a JWT from the authorization server and have it stop working everywhere. The fix has two layers: use short expiry times — five to fifteen minutes for access tokens — so the blast radius of a compromised token is bounded by its TTL. Combine this with Refresh Tokens, which are long-lived but stored server-side and can be revoked. For immediate revocation — such as when a user explicitly logs out or reports a compromised account — maintain a Redis-based token revocation list keyed on jti (the JWT ID claim) with TTL set to the remaining lifetime of the token. Every token validation checks this list. The cost is one Redis lookup per request, which is typically sub-millisecond and acceptable.

Trap 8: Disabling CSRF in Spring Security for the wrong reasons

The trap: disabling CSRF protection across all endpoints without understanding when it is and is not needed, either leaving a vulnerability or causing unnecessary confusion. CSRF attacks exploit the fact that browsers automatically send session cookies with cross-origin requests — so a malicious page can trigger authenticated state-changing requests using the user’s existing session. If your authentication is stateless — using a Bearer token in the Authorization header, not a cookie — the attack is not possible: cross-origin JavaScript cannot set or read the Authorization header for another origin’s requests. For REST APIs using Bearer tokens: http.csrf().disable() is correct and safe. For web applications using session cookies: keep CSRF protection enabled. The common mistake in the wrong direction is disabling CSRF for an application that authenticates via session cookies, thinking it is “stateless” because it uses JWTs — if the JWT is stored in a cookie, CSRF still applies.

Trap 9: Using the OAuth2 Implicit grant

The trap: using the Implicit grant type for SPAs or mobile apps because it is “simpler” — it returns the access token directly in the URL redirect fragment. The problem is that URL fragments are visible in browser history, server access logs, referrer headers sent to third-party resources on the page, and other browser APIs accessible to JavaScript. The access token is effectively visible to any party that can read URLs. The Implicit grant is formally deprecated in OAuth 2.0 Security Best Current Practice and OAuth 2.1. The fix: always use Authorization Code with PKCE for public clients. It is slightly more complex but provides proper security — the authorization code in the redirect is short-lived, single-use, and useless without the code_verifier that only the legitimate client possesses.

Trap 10: Not validating the JWT audience claim

The trap: validating the signature, expiry, and issuer of a JWT but skipping the aud — audience — claim validation. The audience claim specifies which service the token was intended for. If Service A and Service B both trust the same Authorization Server but you do not validate aud, a token issued for Service A can be presented to Service B and will be accepted. This is a token replay attack: an attacker who can obtain a valid token for Service A — perhaps a lower-security service — can replay it against Service B — a higher-security service. The fix: configure your JWT library with the expected audience value for each service, and explicitly reject tokens where aud does not match. In Spring Security OAuth2, this is configured via JwtDecoder with JwtValidators.createDefaultWithIssuer() or by adding a custom JwtClaimValidator for the audience.

Trap 11: N+1 queries in microservices

The trap: replicating the ORM N+1 anti-pattern at the service level — loading a list of entities from one service and then making an individual call to another service for each entity. For example, loading 100 orders from the order service, then calling the inventory service in a loop for each order’s item availability: 100 individual HTTP calls or message round trips. This is 100x the latency of a batched approach and generates 100x the load on the inventory service. The fix: design batch APIs — a single endpoint that accepts a list of IDs and returns a map of results. The call becomes one round trip for all 100 orders. If you own both services, also consider whether some data should be co-located — the inventory snapshot at order time could be denormalized into the order record. In the BFF pattern, the Backend-for-Frontend layer is responsible for orchestrating these batch calls and composing results before returning to the client.

Trap 12: Two microservices sharing a database

The trap: treating a shared database as a simple way for two microservices to share data — Service A and Service B both connect to the same PostgreSQL database and query the same tables. This violates the core microservices principle of independent deployability: if both services share a schema, you cannot change the schema for one without coordinating with the other. You cannot scale their storage independently. You cannot choose different databases for different access patterns. If one service’s heavy queries contend with the other’s, both are affected. The coupling is as tight as a monolith but without the simplicity of a monolith. The fix: each microservice owns its own database schema, inaccessible to other services. Cross-service data needs are fulfilled through well-defined APIs or asynchronous events — if Service B needs data that Service A owns, Service B calls Service A’s API or subscribes to Service A’s domain events and maintains its own local read model.

Trap 13: Circuit breaker half-open timing too short

The trap: setting waitDurationInOpenState — the time the circuit breaker stays OPEN before attempting test calls in HALF-OPEN — to a very short value, like one second, because you want fast recovery. If the downstream service needs 30 seconds to restart and stabilize after an outage, a one-second wait causes the circuit breaker to oscillate rapidly: OPEN → HALF-OPEN → OPEN → HALF-OPEN, firing test requests at the recovering service repeatedly and potentially delaying its recovery. The fix: set waitDurationInOpenState to at least the expected recovery time of the downstream service — typically 10 to 60 seconds for a service restart. Monitor the downstream service’s actual recovery time in production incidents and tune accordingly. Also configure permittedNumberOfCallsInHalfOpenState to allow a few test calls — say three — before fully closing the circuit, to confirm recovery is stable and not just a momentary blip.

Trap 14: Redis cluster cross-slot operations

The trap: using multi-key commands like MGET, MSET, SUNION, or pipelining on keys that hash to different slots in a Redis Cluster setup. In Redis Cluster, a command involving multiple keys is only valid if all keys are in the same hash slot on the same node. If the keys hash to different slots — and therefore different nodes — the command fails with a CROSSSLOT error. This is a common surprise when migrating from a single Redis instance to Redis Cluster. The fix: use hash tags to force related keys to the same slot. A hash tag is a substring enclosed in curly braces in the key name — Redis uses only the text inside the braces to compute the slot. For example, {user:123}:orders and {user:123}:cart both contain the hash tag user:123 and will always land on the same node, making MGET safe. Design your key naming convention with hash tags from the start if you plan to use Redis Cluster.

Trap 15: bcrypt 72-character password truncation

The trap: trusting that bcrypt handles all passwords correctly regardless of length. bcrypt internally operates on a 72-byte input — characters at position 73 and beyond are silently ignored. This means a password like “correct-horse-battery-staple-my-very-long-passphrase” and “correct-horse-battery-staple-my-very-long-DIFFERENT” would produce the same bcrypt hash if their first 72 characters are identical. An attacker who knows the first 72 characters of a user’s password and knows the system uses bcrypt can authenticate successfully with any suffix. The fix is to use Argon2id, which has no input length limit, for all new systems. If you are locked into bcrypt for a legacy system, the mitigation is to pre-hash the password with SHA-256 and Base64-encode the result before passing it to bcrypt: bcrypt(base64(sha256(password))). The SHA-256 output is always 32 bytes — safely within bcrypt’s limit — and two different passwords will produce two different SHA-256 outputs regardless of length.

Trap 16: Kafka exactly-once requiring both producer AND consumer configuration

The trap: configuring enable.idempotence=true on the Kafka producer and believing you have end-to-end exactly-once delivery. Idempotent producer prevents duplicate messages from appearing in Kafka topics due to producer retries — it adds sequence numbers to messages so the broker can detect and discard duplicates from the same producer session. But exactly-once delivery requires the consumer side too: your consumer must read only committed transaction data using isolation.level=read_committed, and your processing plus offset commit must be wrapped in a Kafka transaction using the transactional producer API. Without the consumer-side configuration, your consumer can still process the same message twice if it crashes between processing and committing its offset. The fix: use the full transactional pattern — configure transactional.id on the producer, wrap produce-and-consume in beginTransaction/commitTransaction, and set read_committed on consumers. In Spring Kafka, use KafkaTransactionManager.

Trap 17: Spring @Transactional, @Async, and @Cacheable failing on self-invocation

The trap: calling a @Transactional, @Async, or @Cacheable annotated method from within the same class and observing that the annotation has no effect — transactions do not start, async calls execute synchronously, cache is bypassed. This happens because Spring’s AOP — Aspect-Oriented Programming — implements these annotations by wrapping the bean in a proxy at injection time. When you call this.annotatedMethod() from within the same class, you are calling the raw object directly, bypassing the proxy entirely. The JVM call never goes through the proxy interceptor that implements the transactional, async, or caching behavior. The fix: inject the bean into itself using @Autowired private MyService self — Spring will inject the proxied version — and call self.annotatedMethod(). Alternatively, extract the annotated method into a separate Spring-managed bean and inject that. This is a perennial trap because the code compiles and runs without errors; the annotation just silently does nothing.

Trap 18: Setting linger.ms=0 in a high-throughput Kafka producer

The trap: using the default linger.ms=0 in a production Kafka producer and accepting poor throughput without investigating why. linger.ms controls how long the producer waits for additional messages to arrive before sending a batch — with zero milliseconds wait, the producer sends one message per network request, one batch per message. This is maximally low-latency for individual messages but terrible for throughput and efficiency because each batch has the same per-request overhead — TCP headers, Kafka protocol overhead — regardless of how many messages it contains. At high volumes, this causes thousands of tiny network requests per second. The fix: set linger.ms to a small value like 5 to 20 milliseconds. This allows multiple messages to accumulate into one batch before sending. Combined with batch.size — the maximum batch size in bytes — the producer achieves dramatically higher throughput with only a few milliseconds of added latency. Also set compression.type=snappy or lz4 to reduce network bandwidth for large batches.

Trap 19: Redlock algorithm assuming synchronized clocks

The trap: using the Redlock algorithm for safety-critical distributed locks — such as financial transactions or inventory deductions — and assuming it is as safe as consensus-based locks. Redlock’s safety argument relies on clocks being roughly synchronized across Redis nodes so that time-limited lock validity is meaningful. NTP clock synchronization can experience jumps of tens of milliseconds or more, and long JVM garbage collection pauses — GC stop-the-world events — can cause a thread to believe it holds a valid lock while the lock has already expired on all Redis nodes. In such a scenario, two processes can simultaneously believe they hold the exclusive lock, violating mutual exclusion. The fix: for most business applications — cache invalidation, user-facing features, non-financial operations — Redisson RLock over a single reliable Redis instance is sufficient and the risk window is negligible. For operations where simultaneous dual execution is genuinely catastrophic — financial transactions, critical resource allocation — use a distributed coordination service built on consensus algorithms: ZooKeeper, etcd, or Apache Curator, which provide strong linearizability guarantees that do not depend on clock synchronization.

Trap 20: Exposing sensitive information in health endpoints

The trap: leaving Spring Boot Actuator health endpoints publicly accessible with full details enabled in production. The /actuator/health endpoint by default can expose database connection strings, external service URLs, disk paths, stack traces, environment variable names, and version information — all useful to an attacker for reconnaissance. The default configuration shows summary status only, but management.endpoint.health.show-details=always is commonly set during debugging and never removed. The fix: set management.endpoint.health.show-details=when_authorized so detailed health information is only visible to authenticated and authorized users, typically internal monitoring systems or operators. Also restrict which actuator endpoints are exposed over HTTP: management.endpoints.web.exposure.include should list only the endpoints you specifically need — health and info for a load balancer readiness check, for example — and exclude everything else. Move actuator endpoints to a separate management port on an internal network interface rather than the same port as public API traffic.

Common Mistakes — Section 7:

  • Mixing up 401 and 403 in code and design discussions → 401 = unauthenticated (who are you?), 403 = unauthorized (I know you, you cannot do this). Name them correctly in code and in interviews.
  • Assuming PATCH idempotency without verifying the patch semantics → always ask whether the patch is set-based or increment-based before classifying it.
  • Not profiling OFFSET pagination performance before shipping → add EXPLAIN ANALYZE on your pagination queries in staging; the OFFSET cost only becomes visible at realistic data volumes.
